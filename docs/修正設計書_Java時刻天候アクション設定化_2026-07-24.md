## 1. 目的・背景

- レビュー REV-P2-02 は「`setTime`/`setWeather` を全 trigger で『固定』と表現するが、生成コマンドは `time set` / `weather` を1回出すだけの one-shot であり、event trigger では固定にならない」と指摘した。
- v0.7.2 はこれに対し **interval 専用化**（event trigger では検証エラー）で対処し、「固定」表記を維持した（`apps/web/src/lib/spec.ts:430-435`、`pattern-catalog.ts:160-168`、`java-generator.ts:122-123`）。
- ユーザー確定方針は A 案：**「固定」という継続保証の含意を撤回し「設定」（one-shot）と正しく表記**したうえで、**event trigger でも使用可能**にする（例:「りんごを食べたら昼にする」を表現可能にする）。
- 生成される Minecraft コマンド自体（`time set <value>` / `weather <value>`）は one-shot のまま**不変**。変えるのは「表記（固定→設定）」「使用可能な trigger 範囲（interval 専用→全 trigger）」「利用者向け文書」の3点である。

## 3. 全体方針

- 最小変更。**interval 専用チェックの削除**＋**setTime/setWeather 系の文言置換**＋**利用者向け文書の同期**が中心で、生成ロジックの構造は変更しない（現行は trigger harness と action fragment が分離済みのため、event 許可に新しい生成アーキテクチャは不要）。
- 「固定」は「継続してその状態を保つ」という誤った含意を持つため撤回する。one-shot の事実は「設定（発火時に1回）」という表記と README 注記で正しく伝える。interval trigger では毎周期再適用されるが、それは trigger 側の性質であってアクションの保証ではない、と切り分ける。
- event 許可は制限の**緩和**であり後方互換（v0.7.2 で通っていた interval+setTime はそのまま通る）。

### 3.1 許容差分の定義（REV-P1-01・必須）

現行生成器は `describePack` の結果を `packMcmeta` と `buildReadme` の**双方**へ渡す（`apps/web/src/lib/java-generator.ts:56-62`）。したがって `describeAction` の「固定」→「設定」変更は `pack.mcmeta` の `pack.description` を**必ず変える**。「生成 JSON はバイト一致」という条件は成立しないため、以下のとおり**許容差分（whitelist）**と**不変（バイト一致）対象**を分けて定義する。

**許容差分（この2ファイルのみ変わってよい）**

| path | 許容される差分 |
|---|---|
| `pack.mcmeta` | `pack.description` の文字列のみ（「…に固定」→「…に設定」）。`pack_format` / `min_format` / `max_format` は不変。 |
| `README.txt` | description 行の同変更＋§4 M3 の注記行の追加のみ。 |

**不変（v0.7.2 とバイト一致）**

- action を格納する `.mcfunction` 全て（`load.mcfunction` / `main.mcfunction` / `on_event.mcfunction`）。`time set <value>` / `weather <value>` の行を含め一切変更しない。
- description を含まない生成 JSON 全て（`data/**/recipe/*.json`、`data/minecraft/loot_table/**`、`data/**/predicate/*.json`、`data/**/advancement/*.json`、`data/minecraft/tags/function/load.json`、`assets/minecraft/lang/*.json`、`assets/minecraft/items/*.json`）。

### 3.2 26.2 における setTime の対応範囲（REV-P2-01）

- Minecraft 26.1+ では `/time` が World Clock ベースになり、clock 省略時は**実行 dimension の default clock** を使う。本書は生成コマンドを `time set <value>`（clock 省略）のまま維持するため、**setTime の対応範囲を「オーバーワールドでの実行」に限定して明文化**する。
- これを capability promptLine と生成 README に明記し（§4 M2・M3）、受入条件に **26.2・オーバーワールドでの成功確認**と**非オーバーワールドは対象外である旨の記録**を含める（§8・§9）。
- **setTime と setWeather の制約を混同しない**。World Clock に依存するのは `setTime` のみで、`setWeather` は World Clock の対象外であり版依存の制約を持たない（天候がオーバーワールド系ディメンションでのみ意味を持つのは版非依存の既存バニラ挙動であり、本書では扱わない）。
- 明示 clock 化（`time of minecraft:overworld set <value>`）が必要になった場合は、対象版の分岐を伴うため別スコープで起票する。

## 4. 詳細仕様

### M1: interval 専用制限の撤回（spec.ts）

`validateJavaScript` 内の setTime/setWeather の interval 専用チェックを削除する。

```ts
// 削除対象（apps/web/src/lib/spec.ts:430-435）
// 時刻・天候の「固定」は周期再設定でのみ成立する。イベント系（1回実行）は禁止。
if (
  ["setTime", "setWeather"].includes(a.type) &&
  s.trigger !== "interval"
)
  errors.push("時刻・天候の固定は定期実行トリガーでのみ使えます。");
```

- 削除後、setTime/setWeather は他アクション同様、全 trigger（interval / consumeItem / placedBlock / killEntity / mineBlock / death）で許可される。値未設定チェックは M2 で文言のみ変更して存続する。

## 7. リスクと割り切り

- **description 差分は不可避**（§3.1）。`pack.mcmeta` と `README.txt` は必ず変わるため、これを許容差分として明示し、それ以外の全ファイル一致をテストで固定することで回帰を防ぐ。
- event 許可により「固定」の誤解は解消するが、one-shot のため「ずっとその時刻/天候になる」わけではない。期待外れを避けるため M3 の one-shot 注記を必須とする。
- **26.2・非オーバーワールドでの `time set` は対象外**（§3.2）。promptLine と README で事前に伝えることで「黙って失敗する」状態を避ける。明示 clock 化が必要になれば別スコープ。
- v0.7.2 の interval 専用は**制限緩和**方向のため、既存生成物の互換は保たれる（v0.7.2 で作れたものは v0.7.3 でも作れる）。
- 「固定」の機械的全置換はレシピ検証文言・message 系 promptLine を巻き込み既存テストを壊す。M2 の非対象リストで防ぐ。
- ロールバックは revert 1 コミットで可能。
