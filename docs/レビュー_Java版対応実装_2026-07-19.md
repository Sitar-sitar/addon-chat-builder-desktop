## レビュー結論

**判定: 合格（条件付き）— 実ゲーム受入3種の完了を必須とし、マージ前に P2 1件の対応を推奨する。**

実装は完成版設計に忠実で、設計レビュー（P1×3 / P2×4 / P3×2）の要求事項はすべてコードとテストで確認できた。自動検証（tsc / Vitest 24件 / next build）は本レビューで再実行してすべてグリーン、生成済み ZIP 3種の中身（pack_format 71/55・レシピ形式・schedule 自己再予約・lang 2ファイル同一）も設計 §4.4 と一致した。Bedrock 経路の回帰は認められない。

ただし、Java レシピの **pattern と key の構造整合が未検証**であり、チャット整理（LLM 出力）由来の不整合がそのまま「ゲームに読み込まれないレシピ」として出力され得る。決定論的生成の狙い（P1-02: 無効 zip を正常終了で出さない）を完全にするため、master マージ前の対応を推奨する（下記 P2-01）。

## 指摘事項

| ID | 重大度 | 該当箇所 | 指摘 | 推奨対応 |
|---|---|---|---|---|
| JAVA-IMPL-P2-01 | P2 | `apps/web/src/lib/spec.ts:98-125`（validateRecipe）/ `java-generator.ts:39-53` | Java レシピの **pattern と key の構造整合を検証していない**。(a) pattern 行の同幅（Java は不一致だと "each row must be the same width" でレシピ無効）、(b) 行長 1〜3 文字、(c) pattern 中のシンボルが key に定義されているか、(d) key のシンボルが 1 文字か、を見ていない。チャット整理は LLM 出力なので不整合は現実に起こり得て、その場合「全検証合格・生成成功」なのにゲームで無効なレシピになる。決定論的生成の目的（設計レビューP1-02）が最後の一点で崩れる | `validateSpec` / `specChecks` の java × recipe に (a)〜(d) を追加（設計 §4.1 への追記も design-doc で実施）。境界テストを追加。**master マージ前の対応を推奨**（変更は spec.ts とテストに閉じる） |
| JAVA-IMPL-P3-01 | P3 | `apps/web/src/app/api/build/route.ts` | `/api/chat` は edition を必須化したが、`/api/build` は `body.spec.edition` をそのまま信頼する。validateSpec の edition×kind 整合で実害はほぼ無いが、ロックの対称性がない | 任意対応。chat と同様に `body.edition` を受けて spec.edition と不一致なら 400 にすると契約が揃う |
| JAVA-IMPL-P3-02 | P3 | `apps/web/src/lib/openai.ts:452-457`（normalizeRawSpec） | `intervalSeconds` はモデル出力の number をそのまま採用。非整数（例 60.5）が来ると validateSpec が拒否し、ユーザーには範囲エラー文言だけが出る（整数であることが伝わらない） | 任意対応。normalize 時に `Math.round` で丸めるか、エラー文言へ「整数」を明記 |
| JAVA-IMPL-P3-03 | P3 | `apps/web/src/lib/java-generator.ts:80-94`（buildReadme） | README.txt の組み立てが filter の複合条件で読みにくく、テストもパス存在のみで内容を固定していない（README の回帰検知が弱い） | 任意対応。単純な配列 join へ書き換え、README 全文一致をテストに1件追加 |

## 良い点

- 設計レビューで最も重かった「決定論的生成への転換」が素直な構造で実装されている。`java-generator.ts` は 94 行に収まり、テンプレート・エスケープ（`JSON.stringify`）・バージョン差分（材料表記）の扱いが明快。
- テストが要点を突いている：pack_format の版別完全一致、tellraw の JSON エスケープ（引用符・改行入り）、未知バージョンで zip を作らないこと、edition ロックの上書き、不変条件（specChecks ⇔ validateSpec）の同値性。
- `1.21.5` 行の追加は設計の想定どおり「表への1行追加」で済んでおり、既定値 1.21.7 を変えずにローカル `.env` で選択する運用も設計意図（厳密指定）に合致。実装ログに実機バージョン確認の経緯が残っているのも良い。
- Bedrock 経路は `enforceBedrockRules` フラグで従来検証を完全維持しつつ、重複パス拒否・pack.mcmeta の JSON 検証など共通部分の防御が微増している。
- UI の世代ID破棄が chat / build / select-folder の3経路すべてに一貫適用され、`finally` でのフラグ復帰まで世代確認している。
