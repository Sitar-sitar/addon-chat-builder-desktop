## レビュー結論

設計の方向性（草原テーマ、チャット＋ライブ設計図、`apps/web` 正本への適用）は、現行構成と整合しており実装方針として妥当。

ただし、`canBuild` とライブ設計図の完了判定がずれる点、追加するユニットテストの実行手段が未定義な点は、実装前に設計へ反映した方がよい。P1を解消すれば、UI実装に入れる状態と判断する。

## 指摘事項

| 優先度 | 指摘 | 根拠 | 影響 | 推奨対応 |
|---|---|---|---|---|
| P1 | `canBuild` と `blueprintRows` の完了判定が一致しない | 設計書は `canBuild = validateSpec(spec)` としつつ、`blueprintRows` に `表示名`・`概要`・`名前` などの独自完了条件を持たせている（`design-ui-optimization.md:122-128`, `design-ui-optimization.md:191-202`）。現行 `validateSpec` は `description` を必須にする一方、`item.displayName` と `script.summary` は必須にしていない（`apps/web/src/lib/spec.ts:57-98`）。 | ライブ設計図が「準備OK」でも生成ボタンが無効、または設計図が未完了なのに生成可能、という矛盾が起きる。ステッパー、残数ノート、生成CTAの信頼性が落ちる。 | 単一の判定源を決める。推奨は `description` 行を設計図へ追加し、`item.displayName` / `script.summary` も `validateSpec` 側で必須化すること。必要なら小さな述語関数を `spec.ts` に切り出し、`validateSpec` と `blueprintRows` で共有する。 |
| P1 | 追加予定のユニットテストを実行する npm script / テストランナーがない | 設計書は `apps/web/tests/unit/addon-view.test.ts` を追加し、`npm run lint` → 手動確認の順で完了としている（`design-ui-optimization.md:22`, `design-ui-optimization.md:269-285`）。現行 `package.json` には `test` script と Vitest 等の依存がない（`apps/web/package.json:5-23`）。 | テストファイルを作ってもCI/手元確認で実行されず、将来の回帰検知に使えない。 | 最小構成で `vitest` を devDependency に追加し、`"test": "vitest run"` を定義する。外部依存を増やしたくない場合は、ユニットテスト追加をやめて `npm run lint` の型検査だけにスコープを落とす。 |
| P2 | `suggestedReplies` の件数・長さ・重複制御がプロンプト依存 | 設計書は「0〜4個」「短い日本語」とするが、JSON Schema は配列型のみで `maxItems` 等がなく、戻り値正規化も `recommendedReply` の包含確認までに留まっている（`design-ui-optimization.md:140-150`, `design-ui-optimization.md:156-157`）。 | モデルが長文や5件以上の候補を返した場合、チップUIが崩れたり、推奨チップが複数/不一致に見えたりする。 | Schema に `maxItems: 4` と文字数目安を入れる。戻り値でも `trim`、空文字除去、重複除去、4件上限、`recommendedReply` の包含確認を行う。 |
| P2 | `showChips` と `showReadyCta` が同時表示され得る | 派生条件では `showChips` と `showReadyCta` が独立しており、`canBuild` 到達時に候補返信が残っていると両方成立する（`design-ui-optimization.md:127-128`）。 | 生成できる状態なのに回答候補も並び、利用者が次に何を押すべきか迷う。 | `showReadyCta` を優先し、`showChips = !showReadyCta && ...` とするか、生成CTAの横に「内容を直す」だけを残す仕様にする。 |
| P2 | `next/font/google` は「オフライン安定」という説明とずれる | 設計書は `next/font/google` をオフライン安定のため推奨している（`design-ui-optimization.md:46-48`）。ただし Google Fonts 取得はビルド/開発時のネットワーク要因になり得る。 | ローカルアプリの再ビルド時にネットワーク状態で失敗する可能性があり、半年後の保守再現性が落ちる。 | オフライン再現性を重視するなら、システムフォント＋CSSのみ、またはフォントファイルをリポジトリ内に同梱して `next/font/local` を使う。ネットワーク前提を許容するなら、その前提を設計書に明記する。 |
| P2 | Tabler icon の導入方法が未定義 | 設計書は `ti-check` などの Tabler outline アイコンを指定しているが、現行依存に Tabler 関連パッケージはない（`design-ui-optimization.md:89`, `apps/web/package.json:11-23`）。 | 実装時に CDN、CSS クラス、React コンポーネントのどれを使うか迷い、依存追加やオフライン性の判断がぶれる。 | 依存を増やさない方針ならテキスト記号（チェック、鉛筆、錠）で実装する。アイコン品質を優先するなら `@tabler/icons-react` を追加し、追加理由を設計書に明記する。 |
| P2 | 任意の `/api/open-folder` を実装する場合、WebView2 のトークン付与対象も更新が必要 | 設計書は `/api/open-folder` に `verifyDesktopRequest` を呼ぶとしている（`design-ui-optimization.md:226-228`）。現行 WebView2 ブリッジは `/api/chat` と `/api/build` にだけ `X-Desktop-Token` を付与する（`src/AddonChatBuilder.Desktop/MainWindow.xaml.cs:158-160`）。 | `open-folder` を本当に追加すると、デスクトップアプリ内 fetch でも 401 になり得る。 | MVPでは `/api/open-folder` を実装しない方針を明記する。実装する場合は WebView2 ブリッジのトークン付与対象に `/api/open-folder` を追加するか、HTTP APIではなく WebView2 message で .NET 側へ依頼する。 |
| P3 | 対象設計書のヘッダ・ファイル名が共通ルールと完全には一致しない | 共通ルールは先頭ヘッダに対象アプリ、種別、対象バージョン、前提コミット、ステータス、関連を置くとしている（`D:\my-app2\docs\設計書管理ルール.md:69-81`）。対象設計書は1行ヘッダで、ファイル名も共通規約の日本語プレフィックスではない（`design-ui-optimization.md:1-3`）。 | 後から正本・レビュー・実装後メモの関係を追うときに、状態や前提が少し読みにくい。 | 既存ファイルはリネームしない方針でよい。冒頭に標準ヘッダを追記し、`docs/設計書インデックス.md` で種別と状態を補完する。 |

## 確認事項

- `description`、`item.displayName`、`script.summary` は `.mcpack` 生成の必須条件にするか。UI上だけの見栄え項目にするか。
- ユニットテストのために `vitest` を追加してよいか。依存追加を避けるなら、今回の純粋関数テストは型検査中心に縮小する。
- 「フォルダを開く」は今回のMVPで実装するか。実装する場合は WebView2 bridge も同時変更が必要。

## 良い点

- UI変更のスコープが `apps/web` に閉じており、デスクトップ側のライフサイクル制御と責務が混ざっていない。
- `showSpec` 廃止、常時ライブ設計図、ステッパー、クイック返信の組み合わせは、家族向けの迷いを減らす方向として妥当。
- `verifyDesktopRequest` と `DESKTOP_API_TOKEN` を維持する方針は、現行 WebView2 bridge と整合している。
- 実装順序が純粋関数 → API戻り値 → layout/styles → page の順で、レビュー・差分確認しやすい。

## 実装着手前の推奨順

1. P1-1 の判定整合を設計書へ反映する。
2. P1-2 のテスト実行方針を決める。
3. `suggestedReplies` の正規化、フォント、アイコン、`open-folder` の扱いを設計書へ短く追記する。
4. 設計書B（旧 `addon-chat-builder` 撤去）を先に実施し、編集先を `apps/web` に確定する。
