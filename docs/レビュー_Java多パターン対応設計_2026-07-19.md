## 再レビュー残件

### JAVA-MULTI-REREV-P1-01: Bedrock の「凍結」と共通データ/API変更が両立しない

対象設計書 §5.2 は共通 `AddonSpec` に必須の `unsupportedRequests` と Java 専用フィールドを追加する（対象設計書:287-305）。同時に §5.4・§7 は Bedrock schema/normalize と「normalize 後に Codex へ渡す入力 JSON」を v0.2.0 と完全一致させる（同:612-615,720-721）。

しかし現行 Bedrock 生成は `generateAddonFilesWithCodex(spec)` で受けたオブジェクトを `JSON.stringify({ spec })` して直接送る（`apps/web/src/lib/addon-generator.ts:39-42`, `apps/web/src/lib/openai.ts:238-276`）。共通オブジェクトへ列挙可能な必須 `unsupportedRequests: []` を追加すれば、chat schema を凍結しても Codex 入力 JSON は変わる。対象設計書には v0.2.0 形へ射影する関数も、Bedrock 専用の完全な spec 型もない。

さらに共有 `BuildResult` へ必須 `description` を追加する一方（対象設計書:686-690）、`describePack` の文法は Java 版だけで、Bedrock recipe/item/script が返す値を定義しない。server build gate も Java capability を使う式だけが記され（同:691-694）、Bedrock で `unsupportedRequests` をどう初期化・検証するか決まらない。

**解消条件**:

- `BedrockAddonSpec` と `JavaAddonSpec` を `edition` の discriminated union として完全分離する。少なくとも Bedrock Codex 送信用に `toLegacyBedrockCodexSpec(spec)` を設け、v0.2.0 のキー集合・順序へ明示射影する。
- Bedrock normalize が `unsupportedRequests` を持たない既存 schema から何を返すか、Bedrock build gate ではどの条件を評価するかを固定する。
- `BuildResult.description` の Bedrock 値を定義する（例: 現行 `spec.description` をそのまま返す）。Java の 3 表示一致と混同しない。
- 回帰テストは normalize 結果だけでなく、`generateAddonFilesWithCodex` が構築する user input の `text` を v0.2.0 fixture と全文一致させる。

### JAVA-MULTI-REREV-P1-02: 完全 map と Phase 1〜5 の段階実装が矛盾する

対象設計書は全 Phase の `JavaCapabilityId` と `JAVA_CAPABILITIES` を先行定義し（対象設計書:217-254）、validator/generator を `Record<JavaCapabilityId, ...>` の完全 map にして全 ID の handler 存在を Phase 1 から要求する（同:271-280,733-744）。一方、リリース計画では recipe 拡張、イベント、loot、item model を Phase 2〜5 で順次実装する（同:699-714）。

Phase 1 で完全 map を成立させるには Phase 2〜5 の実 handler まで先行実装するか、未実装 stub を「generator が存在する」と数えるしかない。前者は独立リリースの差分を崩し、後者は完全性テストが生成可能性を保証しない。加えて `capabilitiesForSpec` は script で trigger 1 件＋action 複数件を返すが、各 `GeneratorFn` が同じ `load.mcfunction` / `main.mcfunction` へ何を返し、どの順序で重複なく合成するかが定義されていない。

カタログから作るとする UI スターターも、entry にあるのは `label` と system prompt 用 `promptLine` だけであり（同:245-252,647-648）、ユーザー入力文を決定する `starterPrompt` または選択規則がない。

**解消条件**:

- 次のどちらかを設計で選ぶ。
  - Phase 1 で Phase 1〜5 の全 handler を実装し、後続 Phase は gate 開放と受入だけにする。この場合、Phase 1 の変更量・テスト・ロールバック計画をその前提へ直す。
  - 各 Phase で catalog ID と handler を同時追加する。完全性は「そのリリースに含む ID 集合」に対して検査し、将来 ID を未実装 stub で埋めない。
- `GeneratorFn` の入力・出力を型で固定する。script trigger は harness、action は command fragment、condition は wrapper、というように合成順・ファイル所有者・重複時エラーを定義する。
- UI スターターをカタログ連動させるなら `starterPrompt` を entry へ追加するか、`promptLine` から生成する決定論規則を明記する。

### JAVA-MULTI-REREV-P1-03: 26.2 の time_check テンプレートが確定していない

対象設計書は 26.1+ の版差として `timeCheckClock: true` を持つ（対象設計書:657-672）が、predicate は全版で clock なしの `period` / `value` 形式を出すと固定し、26.2 で無効なら実装開始時に分岐を追加するとしている（同:453-471,833-843）。この列は現状どのテンプレートにも使われず、「版差は表の列で吸収する」「26.2 対応を Phase 1 で完了する」という契約を満たさない。

Minecraft 26.1 の公式 technical changes は `minecraft:time_check` に World Clock ID の `clock` フィールドを追加したことを明記している。今回の必須受入は 26.2 の「夜間に暗視を定期付与」なので、互換性を実装時の成否確認へ残すと Phase 1 の固定テンプレート自体が未確定となる。

**解消条件**:

- `timeCheckClock` が true の版で出す JSON を設計本文へ固定する。想定するなら `"clock":"minecraft:overworld"` を night/day の各 `minecraft:time_check` へ追加し、false の版は現行形とする。
- 1.21 / 1.21.4 / 1.21.5 / 1.21.7 / 26.2 の day・night predicate 全文 fixture を受入条件へ置く。
- 公式仕様または実機確認で想定と異なる場合は、実装を始める前に設計本文を改訂して再々レビューする。「実装側で分岐を決める」逃げ道は残さない。

公式根拠: [Minecraft Java Edition 26.1 technical changes](https://www.minecraft.net/nl-nl/article/minecraft-java-edition-26-1) / [Minecraft Java Edition 26.2 technical changes](https://www.minecraft.net/it-it/article/minecraft-java-edition-26-2)

### JAVA-MULTI-REREV-P2-01: objective hash の非衝突保証が成立しない

対象設計書は SHA-1 の先頭 11 hex（44 bit）を使い「決定論的に衝突しない」とする（対象設計書:375-379）。有限長へ短縮する以上、異なる入力が同じ 44 bit になる可能性は数学的にゼロにできず、「類似 namespace のテスト」でも一般の非衝突は証明できない。

**解消条件**:

- 表現を「衝突確率を十分低くする」へ訂正し、同一 pack 内で生成する objective 群の重複検出を実装契約へ追加する。
- 重複検出時は build を明示エラーにするか、16 文字制約内で決定論的な再 hash/salt 規則を定義する。

## 初回レビュー結論

**判定: 着手不可（No-Go）**

パターンカタログ、決定論的生成、段階リリースという方向性自体は妥当である。しかし、現設計は「対応能力を一元化して意図の黙殺を防ぐ」という中核目的を、型・検証・生成・UI・版差の実装契約まで閉じられていない。

このまま実装すると、少なくとも次の破綻経路が残る。

1. Phase 1 の `script.interval` が有効なだけで、Phase 4 の `setTime` / `setWeather` まで生成可能になる、または実装側に別の手書き分岐が増え、カタログが単一の真実源にならない。
2. 新 `ScriptTrigger` から既存 Bedrock の `itemUse` / `blockBreak` が消えるため、互換変換を実装できず Bedrock script を回帰させる。
3. 未対応要望を `unresolvedQuestions` に記録しても生成可否は止まらず、さらに actions の超過を黙って切り捨てるため、「要望の一部を実現したことにする」事故を再発させる。
4. 2026-07-19 時点の最新安定版 26.2 を追加しようとすると、データパック形式 107.1 と `min_format` / `max_format` の minor 形式を現行の `number` 型で正確に表せない。
5. Phase 4 のカスタム風アイテムと Phase 5 の item model swap は、必要な spec 項目が設計に存在せず生成不能である。

P1 7件を設計本文へ反映し、P2 の実装契約も閉じた後に再レビューすること。P1 を未解消のまま Phase 1 へ着手してはならない。

## 初回詳細指摘

### JAVA-MULTI-REV-P1-01: カタログが単一の真実源になっていない

設計 §3 は、仕様項目・検証・プロンプト・UI・生成テンプレートを 1 エントリへ集約すると宣言する（対象設計書:82-88）。しかし §5.1 の `JavaPattern` が持つのは `id` / `kind` / `label` / `promptLine` / `requires` / `phase` だけであり（同:189-210）、次を表現できない。

- pattern を spec から識別する規則
- action ごとの利用可能 phase と版要件
- 必須入力、既定値、検証、設計図行、スターター文言
- 生成関数または生成テンプレートへの dispatch
- 現在リリース済みの phase

特に `script.interval` は Phase 1 の単一 ID だが、同じ interval に載る `setTime` / `setWeather` は Phase 4 である。`enabledJavaPatterns(rule)` が `script.interval` を返しただけでは、Phase 4 action を拒否できない。関数引数にも release phase がなく、カタログへ Phase 1〜5 を先行登録した際に何を有効とするか決まらない。§9 の「phase ガードを無効化してロールバック」も、ガードの正本がないため実行不能である。

**解消条件**:

- trigger pattern と action capability を分離するか、`script.interval.effect` 等の原子的 capability として表現する。
- `releasedJavaPhase` をビルド時定数など 1 箇所に定義し、`enabledJavaCapabilities(rule, releasedPhase)` とする。
- `capabilitiesForSpec(spec)` を定義し、validate / prompt / UI / generator が同じ capability ID を参照する。
- catalog entry に入力・検証・表示・starter・generator dispatch を持たせるか、「カタログは可用性と文言だけの正本」と責務を縮小して、別の正本を明示する。
- 「全 enum 値に catalog/validator/generator が存在する」「未リリース capability は全経路で拒否する」網羅テストを追加する。

### JAVA-MULTI-REV-P1-02: Bedrock の既存イベントを表現できない

現行 `AddonSpec.script.event` は `itemUse | blockBreak | interval` であり、Bedrock Codex 生成へそのまま渡る（`apps/web/src/lib/spec.ts:22-27`, `apps/web/src/lib/openai.ts:238-315`）。新設計の `ScriptTrigger` は `interval | consumeItem | placedBlock | killEntity | mineBlock | death` で、`itemUse` と `blockBreak` が消えている（対象設計書:225-255）。

§5.2 は `script.trigger → 旧 event` とだけ記すが（同:281-282）、`consumeItem` は一般の `itemUse` と同義ではなく、`mineBlock` も Bedrock Script API の汎用 `blockBreak` と同じ契約ではない。どの値をどう戻すか定義できない。strict schema を Java/Bedrock で共有したままなら、既存 Bedrock の会話仕様そのものが失われる。

**解消条件**:

- 推奨: `BedrockScriptSpec`（既存 event/message）と `JavaScriptSpec`（trigger/condition/actions）を edition 別に分離し、Raw chat schema と normalize も edition 別にする。
- 共通形を維持する場合は `itemUse` / `blockBreak` を union に残し、edition×trigger の互換表を明記する。Java 専用 trigger を Bedrock へ曖昧変換しない。
- 既存 Bedrock recipe / item / script 3 種について、normalize 後に Codex へ渡す JSON の全文一致テストを追加する。

### JAVA-MULTI-REV-P1-03: 未対応・未解決でも生成でき、超過 action を黙殺する

設計は未対応要望を `unresolvedQuestions` へ残すが（対象設計書:408-417）、生成可否へ反映する規則がない。現行 `validateSpec` / `specChecks` は `unresolvedQuestions` を見ず（`apps/web/src/lib/spec.ts:71-95,238-350`）、UI は `validateSpec(spec).length === 0` だけで生成ボタンを出す（`apps/web/src/app/page.tsx:55-65`）。strict schema は使わない種類にも安全な既定値を要求するため、モデルが「新モブは未対応」と記録しつつ、既定の recipe や message を valid に埋めれば、ユーザーは無関係なパックを生成できる。

さらに §5.4 は `actions` を最大 3 件へ切り詰めるとしている（対象設計書:414-417）。4件依頼したユーザーの要望を通知せず削るため、「未対応部分を黙って別表現へ落とす」という今回の根本事故を別の形で再導入する。

**解消条件**:

- build readiness を `validateSpec(...) が空 && unresolvedQuestions が空 && unsupportedRequests が空` と定義する。会話中の不足質問と原理的未対応は別フィールドに分けてもよい。
- `/api/build` でも同じ条件を必ず再検証し、client の表示だけに頼らない。
- action は JSON schema の `maxItems: 3` と prompt で制約し、ユーザー要求が4件以上なら「3件へ減らす質問」を unresolved に残す。normalizer で黙って `slice(0, 3)` しない。
- 「新モブ追加」「4 actions」「未入力項目あり」で build が無効となるテストを追加する。

### JAVA-MULTI-REV-P1-04: 最新版の pack format と版差を表現できない

設計の `JavaVersionRule` は `datapackFormat: number` / `resourcepackFormat: number` と `mcmetaFormat: legacy | minMax` を持つ（対象設計書:429-452）。しかし公式仕様では Java 1.21.9 から pack version に minor version が導入され、`min_format` / `max_format` の完全形は整数2要素の配列で表す。2026-07-19 時点の最新安定版 26.2 の Data Pack version は **107.1**、Resource Pack version は **88.0** である。

JavaScript の `107.1` という小数を JSON の `min_format` に書くことは、公式が定義する「整数または整数配列」と一致しない。`[107, 1]` を保持できる型が必要である。また最新系列へ至る 26.1 では `minecraft:time_check` に `clock` が追加され、cooking 4型の result count も初めて受理される。`plainIngredients` / `componentSyntax` / `mcmetaFormat` の3差分だけでは、最新版を「行追加だけ」で吸収できない。

**解消条件**:

- `PackFormat = number | readonly [major: number, minor: number]` 等を定義し、`packMetadata` を legacy/new 世代ごとの完全な生成契約にする。
- 2026-07-19 時点の最新安定キー（26.2）と 107.1 / 88.0 を設計に固定し、実装時調査へ先送りしない。後日更新する場合は別行として追記する。
- `timeCheckClock`、`cookingResultCount` など実際にテンプレート分岐へ必要な capability を表へ追加する。単に `componentSyntax` へ押し込めない。
- 全対応版について `pack.mcmeta` 全文と、版差を持つ pattern の生成全文をテストする。

公式根拠: [Java 1.21.9 の Pack Metadata](https://www.minecraft.net/en-us/article/minecraft-java-edition-1-21-9) / [Java 26.1 の technical changes](https://www.minecraft.net/en-us/article/minecraft-java-edition-26-1) / [Java 26.2 の technical changes](https://www.minecraft.net/en-us/article/minecraft-java-edition-26-2)

### JAVA-MULTI-REV-P1-05: Phase 4/5 の入力モデルが欠落している

Phase 5 は `resourcepack.itemModelSwap` を掲げるが、§5.2 の `resourcepack` 型は更新されておらず、現行 `langEntries` 以外に `targetItem` / `sourceItem` や resourcepack pattern type がない（現行 `apps/web/src/lib/spec.ts:28-30`、対象設計書:217-279,397-402）。この形では `resourcepack.lang` と `resourcepack.itemModelSwap` を識別できず、生成パスと参照モデルを決定できない。

同様に Phase 4 の「カスタム風アイテム」は、名前・lore・enchantment・components・適用 recipe を表す spec がなく、固定テンプレートもない（対象設計書:154,442-466）。`componentSyntax` 列だけでは入力値を作れない。

**解消条件**:

- `resourcepack.pattern: "lang" | "itemModelSwap"` と item model swap の必須2 IDを spec/schema/validator/UI/descriptionへ追加する。
- カスタム風アイテムを本当に Phase 4 に含めるなら、対応 component の有限集合、spec、版別 JSON、検証、全文一致 fixture を定義する。定義できないなら Phase 4 と「Phase 1〜5 全実施」から外し、別設計にする。
- Java 1.21.4 の公式仕様どおり `/assets/[namespace]/items/[path].json` と model reference の対応をテストする: [Java 1.21.4 Item models](https://www.minecraft.net/en-us/article/minecraft-java-edition-1-21-4)

### JAVA-MULTI-REV-P1-06: イベント生成が固定テンプレートになっていない

§3 は全コマンド・JSONを §5.3 に全文定義するとする（対象設計書:82-88）。しかしイベント駆動は `placedBlock → ... + block 条件`、`killEntity → ... + entity 条件`、`<actions を @s 向けに展開>` という説明だけで、JSON key と値の完全形がない（同:340-370）。

また `condition` は全 trigger 共通の spec だが、イベント時に predicate を action の前へ付けるのか、advancement criteria に組み込むのか、条件不成立でも revoke/reset するのかが未定義である。たとえば「夜にリンゴを食べたら回復」は、実装者の判断によって昼に発火したり、昼の達成状態が残って夜に遅延発火したりし得る。

**解消条件**:

- consume/placed/kill の advancement JSONを、それぞれ完全な JSON fixture として設計へ記載する。
- event × condition は `on_event` 内で各 action のみ predicate guardし、末尾 revoke/reset は常に実行する、等の状態遷移を明記する。
- interval と event で各 action の selector、実行位置、global action の扱いを表にする。
- trigger 5種 × condition 5種の代表境界（少なくとも always と条件付き）をテストする。

### JAVA-MULTI-REV-P1-07: 前提リリースの受入ゲートが閉じていない

現 `docs/実装ログ.md:20-29` は v0.2.0 を完了扱いにしつつ、Java 1.21.5 実ゲーム3種（recipe / interval / lang）が未確認であると明記する。先行設計も実ゲーム3種を必須ゲートとしていた（`機能設計書_Java版対応_2026-07-18.md:366-386`）。

今回の Phase 1 は既存 script、pack metadata、description、OpenAI schema をまとめて変更するため、ベースラインが未確認のままでは「もともと壊れていた」のか「Phase 1 で壊した」のか判別できない。段階リリース原則とも矛盾する。

**解消条件**:

- Phase 1 着手前に v0.2.0 の 1.21.5 実ゲーム3種を完了し、実装ログへ結果を記録する。
- 不合格なら v0.2.0 修正を別フェーズで先に閉じる。未実施のまま開始する場合はユーザーの明示的 waiver と、少なくとも生成 ZIP の既知状態・期待結果をベースライン fixture として固定する。

### JAVA-MULTI-REV-P2-01: 版依存検証のデータフローがない

設計は `validateSpec` が `enabledJavaPatterns(rule)` を参照するとするが（対象設計書:283-293）、`rule` をどう受け取るか定義しない。現行 UI は `validateSpec(spec)` を client で同期実行し、対象版は後から `/api/config` で文字列だけ取得する（`apps/web/src/app/page.tsx:48-75`）。server build は環境変数から別途版を解決する。

**解消条件**:

- `validateSpec(spec, javaCapabilities)` / `specChecks(spec, javaCapabilities)` の契約を定義する。
- `/api/config` は解決済み version と client に必要な capability だけを返す。server `/api/build` は同じ `resolveJavaVersion()` から独立に再検証する。
- config 取得前・取得失敗時は Java の build/chat starter を無効にする。
- client と server の capability parity テストを追加する。

### JAVA-MULTI-REV-P2-02: recipe の resultCount が黙殺される

共通 recipe spec は `resultCount: 1〜64` を持つが（対象設計書:258-272）、cooking テンプレートは count を出さず、smithing は count=1 を固定する（同:374-384）。入力では2個以上を受け、成果物では1個に変わる経路である。さらに設計は cooking result の count 可否を「版ごとに実装時確定」とし、「未確定なし」と矛盾する。

公式仕様では item stack の count は 1.20.5 から一般の結果オブジェクトに存在する一方、smelting/blasting/smoking/campfire_cooking が count を受理する変更は 26.1 で明示されている。旧対応版と最新版で同じ扱いにはできない。

**解消条件**:

- recipeType×version ごとの resultCount 可否を capability 表へ固定する。
- 非対応版は `resultCount === 1` を検証し、対応版だけ `{id,count}` を出す。smithing を常に1固定するなら spec validation と UI も1へ固定する。
- 1 / 2 / 64 の境界と、旧版で2を拒否するテストを追加する。

公式根拠: [Java 1.20.5 Item Stack format](https://www.minecraft.net/en-us/article/minecraft-java-edition-1-20-5) / [Java 26.1 Recipes](https://www.minecraft.net/en-us/article/minecraft-java-edition-26-1)

### JAVA-MULTI-REV-P2-03: description の3表示統一が API まで閉じていない

設計は `describePack` を pack.mcmeta / README / ビルド完了メッセージへ使うが（対象設計書:299-304）、現行 `BuildResult` は `packPath` / `files` だけで、page は独自文言を組み立てる（`apps/web/src/lib/addon-generator.ts:16-19,49`, `apps/web/src/app/page.tsx:20-23,154-158`）。変更一覧にも `addon-generator.ts` と `/api/build` response の更新がない（対象設計書:480-492）。

また `describePack` の文法は interval×effect の1例だけで、複数 actions、event、recipe、loot、resourcepack の決定規則がない。

**解消条件**:

- `BuildResult` に `description` を追加して server 生成値を client へ返すか、3表示すべてを同一 pure module で生成する明示契約を置く。
- kind/trigger/condition/action/複数actionの語順・上限長・未知ID表示を文法表で定義する。
- 3出力が同一文字列であることを integration test で固定する。

### JAVA-MULTI-REV-P2-04: ID と objective 名の検証契約が不足する

設計は「`minecraft:` 形式検証」とだけ記すが、現行検証は多くが `startsWith("minecraft:")` 相当である（`apps/web/src/lib/spec.ts:131-135`）。Phase 4/5 では ID の path 部分を生成ファイルパスへ直接使うため、許容文字、slash、`.` / `..`、namespace 除去方法を定義しないと、生成後の共通パス検証で落ちるか意図しない場所を指す。

scoreboard 方式も objective 名を `<ns>_ev` とするだけで（対象設計書:356-370）、対象版の command 制約に収める長さ・文字・衝突回避規則がない。現行 namespace は長さ上限を持たない（`apps/web/src/lib/spec.ts:83-85`）。

**解消条件**:

- Minecraft resource location の正規表現を1箇所に定義し、ID用途とファイル path 用途を分ける。生成 path へ使う値は traversal 不可を明示する。
- objective は namespace から直接作らず、固定prefix＋短い決定論 hash など、対象版制約内で衝突しない関数を定義する。
- 長い namespace、不正 path、類似 namespace の衝突テストを追加する。

### JAVA-MULTI-REV-P2-05: 対応版×パターンの回帰網が不足する

§10 は「新パターンごとの代表 spec」と世代別 pack.mcmeta を挙げるが（対象設計書:512-526）、各パターンを全対応版で生成する契約ではない。今回の設計は ingredients、item models、components、pack metadata、predicate、recipe count と版差が複数軸に増えるため、代表1版だけの fixture では別版の破損を検出できない。

**解消条件**:

- `for each JAVA_VERSIONS × each enabled capability` の生成成功/拒否マトリクスを自動テストする。
- 版差のないファイルは同一性、版差のあるファイルは期待 fixture を固定する。
- 実ゲーム受入を「既定版＋導入版」に絞る場合でも、広告する全対応版は最低限 pack metadata とデータ schema の静的 fixture を持つ。
- Bedrock 回帰は normalize だけでなく build 経路の representative 3種を維持する。

### JAVA-MULTI-REV-P3-01: 設計の確定状態と履歴が一致しない

§12 は「未確定なし」とする一方（対象設計書:548-557）、本文には `componentSyntax` の実値、最新版の pack format、cooking result の count 可否を実装時確定として残す（同:378-382,442-447）。変更履歴も「ユーザー確定5点」と書くが、表は6点である（同:566）。

**解消条件**: 実装判断が必要な項目を設計で確定するか「未確定事項」へ戻し、着手ゲートを明記する。変更履歴の件数を6へ直す。

## 良い点

- Java生成をLLMへ戻さず、決定論的テンプレートを維持する判断は正しい。
- 「未対応を未対応と明言する」「自由記述 description を成果物へ載せない」という事故分析は適切である。
- Phase 1〜5を分割し、各段階に実ゲーム受入と実装ログ締めを置く運用は、設計契約が閉じれば有効である。
- Java 1.21.4 item model definitions、1.21.9 pack metadata 世代、recipe/predicate の版差を同じ `JavaVersionRule` 系へ集約しようとする方向は保守しやすい。
- loot table 上書きでシルクタッチ・幸運が失われること、ID実在確認をしないことを利用者向けに明示する姿勢は妥当である。

## 推奨修正順

1. P1-07 の v0.2.0 実ゲーム受入を先に閉じる。
2. Bedrock/Java の script spec を分離し、P1-02 の既存契約を保存する。
3. capability catalog を原子的に再設計し、release phase・spec→capability・action gate を定義する。
4. unresolved/unsupported を build gate に接続し、黙示的な切り詰めを廃止する。
5. 26.2 を含む pack format と template capability の版差表を確定する。
6. Phase 4/5 の不足 spec と、event JSON/condition 状態遷移を全文定義する。
7. version data flow、recipe count、description API、ID/objective、全版マトリクスを閉じる。
8. 元設計書と変更履歴へ反映後、再レビューで Go/No-Go を判定する。
