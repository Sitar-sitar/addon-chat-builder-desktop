## 1. 目的・背景

現行再評価レビュー（No-Go）の要点は次の 8 件。いずれも「未対応の意図を別パターンへ黙って劣化させない」という本系統の出発点を、実行時境界・リリース状態・文書整合の面で徹底しきれていないことに起因する。

| ID | 重要度 | 要旨 |
|---|---|---|
| REV-P1-01 | P1 | `/api/build` の未知判別値（`kind` / `recipeType` / `resourcepack.pattern`）が fail-close せず、既知パターンへ黙ってフォールスルーする |
| REV-P1-02 | P1 | Phase 1〜5 の実ゲーム受入前に `RELEASED_JAVA_PHASE = 5` で全公開。かつ phase 0 が v0.2.0 の interval×message を含まず「値を下げるだけの v0.2.0 ロールバック」が成立しない |
| REV-P2-01 | P2 | `JavaVersionRule`（第3引数 `rule`）省略を許す fail-open 契約。Java 検証で rule 欠落時も生成方向へ進み得る |
| REV-P2-02 | P2 | setTime/setWeather を全 trigger で「固定」と表現するが、event trigger は 1 回実行で固定にならない。26.x World Clock 境界も未確定 |
| REV-P2-03 | P2 | 生成 README のデータパック導入先が曖昧（グローバル datapacks 誤誘導）で、既発の不動作を再発させ得る |
| REV-P2-04 | P2 | 版×capability の「期待ファイル／内容 fixture」が未実装。matrix test は生成成功しか見ていない |
| REV-P2-05 | P2 | 設計書の「generator 完全 map 契約」と実装の「coverage 表＋内容テスト契約」が矛盾したまま両方が正本に読める |
| REV-P3-01 | P3 | 機能設計書ヘッダ・変更履歴・インデックス・実装ログが現行事実と非同期。実装ログに重複見出しの疑い |

本書はこの 8 件を、コード修正（REV-P1-01・P1-02・P2-01・P2-02・P2-03・P2-04）と文書修正（REV-P2-05・P3-01）に分けて解消する。

## 3. 全体方針と waiver

- いずれも局所修正で、新機能追加はしない。外部ライブラリ（zod 等）は導入しない（KISS・依存を増やさない）。fail-close は既存の hand-written 検証を「既知 enum の明示チェック＋未知値の明示エラー/throw」へ強化して実現する。
- **fail-close の二段防御方針（REV-P1-01）**:
  - 第1層は `validateSpec` を経由する build gate（client 表示・server `/api/build`）。ここは throw ではなく**エラー文字列**を返し、reactive な UI 検証を壊さない。
  - 第2層は `capabilitiesForSpec` / `generateJavaFiles` の **throw**。検証を迂回した経路（直接 API 呼び出し・古い UI・壊れた LLM 応答が normalize をすり抜けた場合）でも、既知パターンへ化けた成果物を生成しない。
  - `/api/build` に別途の request schema parser（zod 等）は**置かない**。第1層（buildPack 内 validateSpec）が未知値でエラーを返し、既存の route catch が 400 を返すため、依存を増やさずに「未知値→400・出力なし」を満たす。第2層が独立層として重ねられる。
- **段階リリース状態の明示 waiver（REV-P1-02）**: ユーザーの明示指示「受け入れ試験は後日実施でよい」に基づき、`RELEASED_JAVA_PHASE = 5` を据え置いたまま Phase 1〜5 の実ゲーム代表ケースを**未受入のまま先行公開**する。
  - リスク: 実機で未検証のテンプレート（暗視・精錬・鍛冶・食事イベント・loot/時刻固定・itemModelSwap）が生成可能。生成物は自動テスト（型・生成内容一致・gate）で担保されるが、Minecraft 実機での受理は未確認。
  - 緩和: ID 実在確認をしない割り切りどおり、無効 ID/未受理構文は該当処理が無効になるだけでパックはクラッシュしない設計。受入は後日 §機能設計書 §6 の代表ケースで実施し、不合格時は該当 Phase の修正設計書で回収。
  - この waiver は本書 §3・実装ログ v0.7.2 フェーズ・機能設計書変更履歴に記録する。
- **「値を下げるだけのロールバック」の成立（REV-P1-02）**: phase 0 を v0.2.0 実出荷（shaped / lang / interval×message）と一致させることで、`RELEASED_JAVA_PHASE = 0` が v0.2.0 相当へ正しく縮退する。これをテストで固定して主張を裏付ける。

## 4. 詳細仕様

### REV-P1-01: 未知判別値の fail-close（spec.ts / pattern-catalog.ts / java-generator.ts）

**既知 enum 定数（`spec.ts` に 1 箇所）**

```ts
export const JAVA_KINDS = ["recipe", "script", "resourcepack", "loot"] as const;
export const JAVA_RECIPE_TYPES = [
  "shaped", "shapeless", "smelting", "blasting", "smoking",
  "campfire_cooking", "stonecutting", "smithing_transform",
] as const;
export const RESOURCEPACK_PATTERNS = ["lang", "itemModelSwap"] as const;
```

**第1層：`validateJava`（throw しない）**

`validateJava` の冒頭で判別値の既知性を検査し、未知なら明示エラーを push して early return する（`capabilitiesForSpec` を未知値で呼ばない）。

```ts
function validateJava(spec, capabilities, rule, errors): void {
  if (!rule) { errors.push("Java版の設定を取得できていません。生成できません。"); return; } // REV-P2-01
  if (!JAVA_KINDS.includes(spec.kind)) { errors.push(`未対応の種類です: ${spec.kind}`); return; }
  if (spec.kind === "recipe" && spec.recipe &&
      !JAVA_RECIPE_TYPES.includes(spec.recipe.recipeType)) {
    errors.push(`未対応のレシピ種類です: ${spec.recipe.recipeType}`); return;
  }
  if (spec.kind === "resourcepack" && spec.resourcepack &&
      !RESOURCEPACK_PATTERNS.includes(spec.resourcepack.pattern)) {
    errors.push(`未対応のリソースパック種別です: ${spec.resourcepack.pattern}`); return;
  }
  for (const id of capabilitiesForSpec(spec))
    if (!capabilities.includes(id)) errors.push(`現在のJava版では未対応の機能です: ${id}`);
  // 以降は現行の kind 別検証を維持
}
```

- 未知値検出時に early return するため、後続の kind 別検証（`validateJavaRecipe` 等）で未知値を触らない。
- 判別値の TypeScript 型（`JavaRecipeSpec["recipeType"]` 等）は狭いままだが、`includes` は string 比較なので実行時の未知値（型を無視した JSON）を検出できる（`.includes(x as string)` 相当の実装で吸収）。

**第2層：`capabilitiesForSpec` / `generateJavaFiles`（throw）**

`capabilitiesForSpec` の catch-all（未知 recipeType→smithing、未知 pattern→lang、未知 kind→resourcepack）を排し、未知値は throw する。

```ts
export function capabilitiesForSpec(spec: AddonSpec): JavaCapabilityId[] {
  if (spec.edition !== "java") return [];
  if (spec.kind === "recipe") {
    const t = spec.recipe?.recipeType ?? "shaped";
    if (t === "shaped") return ["recipe.shaped"];
    if (t === "shapeless") return ["recipe.shapeless"];
    if (["smelting","blasting","smoking","campfire_cooking"].includes(t)) return ["recipe.cooking"];
    if (t === "stonecutting") return ["recipe.stonecutting"];
    if (t === "smithing_transform") return ["recipe.smithing"];
    throw new Error(`未対応のレシピ種類です: ${t}`);
  }
  if (spec.kind === "script") { /* 現行どおり（javaScript 未設定は [] のまま） */ }
  if (spec.kind === "loot") return ["loot.blockDrop"];
  if (spec.kind === "resourcepack") {
    if (spec.resourcepack?.pattern === "itemModelSwap") return ["resourcepack.itemModelSwap"];
    if (spec.resourcepack?.pattern === "lang" || !spec.resourcepack) return ["resourcepack.lang"];
    throw new Error(`未対応のリソースパック種別です: ${spec.resourcepack.pattern}`);
  }
  throw new Error(`未対応の種類です: ${spec.kind}`);
}
```

- `resourcepack` 未設定（`!spec.resourcepack`）は現行どおり `resourcepack.lang` にフォールバックする（LLM 応答途中の既定挙動を変えないため。未知“値”だけを弾く）。
- `generateJavaFiles` は先頭で `capabilitiesForSpec(spec)` を呼ぶため、未知値なら同 throw が伝播する。加えて生成器内の kind/recipeType 分岐にも最終 `else` で throw を置き、直接生成経路の防御を二重化する。

**テスト（REV-P1-01）**: `build-pack.test.ts` に未知 3 種（`kind:"__x"` / `recipeType:"__x"` / `resourcepack.pattern:"__x"` を型キャストで注入）を追加し、`buildPack` が明示エラーで throw し出力ディレクトリが空であることを固定。`spec.test.ts` で `validateSpec` が対応する日本語エラーを返すこと、`java-generator.test.ts` で `generateJavaFiles` が throw することを固定。

### REV-P2-01: `rule` を Java 検証の必須 context 化（spec.ts / page.tsx / addon-view.ts）

- `validateSpec(spec, capabilities?, rule?)` のシグネチャは維持（Bedrock は rule 不要のまま）。
- `validateJava` 冒頭で `if (!rule) { errors.push("Java版の設定を取得できていません。生成できません。"); return; }`（REV-P1-01 の第1層に統合）。これにより **rule 欠落の Java 検証は常に不可**。
- server（`buildPack`）は `resolveJavaVersion().rule`（未知版は throw 済み）を必ず渡すため影響なし。
- client（`page.tsx`）: `javaVersionRule` は `/api/config` 取得まで `undefined`。`javaGateOpen` の条件に `!!javaVersionRule` を追加し、取得前は生成不可を明示する。

```ts
const javaGateOpen =
  spec.edition !== "java" ||
  (!!javaTargetVersion && !!javaVersionRule &&        // ← javaVersionRule を追加
    spec.unresolvedQuestions.length === 0 &&
    spec.unsupportedRequests.length === 0);
```

- `specChecks` / `blueprintRows` は `validateSpec` 経由のため追加変更不要（rule 欠落時は検証エラーが立つ）。

**テスト（REV-P2-01）**: `spec.test.ts` に「26.2 で有効な cooking（resultCount=2）spec を `rule=undefined` で `validateSpec` すると『設定を取得できていません』で拒否される（caps に cooking を含めても）」を追加。

### REV-P2-02: setTime/setWeather を interval 専用化（spec.ts / pattern-catalog.ts）

- **検証（`validateJavaScript`）**: action.type が `setTime` または `setWeather` で `s.trigger !== "interval"` のとき
  `errors.push("時刻・天候の固定は定期実行トリガーでのみ使えます。")`。
- **意味づけ**: interval は intervalSeconds ごとに再設定するため、時刻/天候の「固定」はドリフト ≤ intervalSeconds の近似固定として成立する。event 系（1 回実行）は「固定」にならないため禁止し、矛盾を構造的に解消する。
- **文言整合**: `describePack` の「時刻を〜に固定 / 天候を〜に固定」は interval 専用となり整合（変更不要）。`pattern-catalog.ts` の setTime/setWeather の `promptLine` に「定期実行トリガーでのみ使用可」を明記。
- **26.2 World Clock（`time set`）**: 生成器の `time set <value>` / `weather <value>` は現行の単純出力を維持（overworld 想定）。26.1+ の World Clock 差（clock 省略時の default dimension 依存）は本書では厳密化せず、**§機能設計書 §13 実装時検証ゲートの項目**として「26.2 実機で setTime/setWeather の受理を確認し、乖離時は設計改訂」を追記して受入で確定する。setTime/setWeather は phase 4・実ゲーム受入 waiver 対象のため、この先送りは許容範囲。

**テスト（REV-P2-02）**: `spec.test.ts` に「event trigger（例 consumeItem）＋ setTime アクションは『定期実行トリガーでのみ』で拒否」「interval ＋ setTime は許可」を追加。matrix の `specFor("script.action.setTime")` は既定で interval トリガーを使うため既存整合。

### REV-P2-03: README 導入先の確定文（java-generator.ts）

`buildReadme` の導入行を、パック種別で分岐した確定文へ置き換える。「対象ワールドまたはMinecraftの datapacks フォルダへ配置」は削除。

- **datapack**（recipe/script/loot）:
  ```
  導入: このzipファイルを解凍せずそのまま、対象ワールドの
        %APPDATA%\.minecraft\saves\<ワールド名>\datapacks\ フォルダへ入れてください。
        ゲーム内で /reload を実行し、/datapack list enabled に本パックが表示されれば有効です。
  ```
- **resourcepack**（lang/itemModelSwap）:
  ```
  導入: このzipファイルを解凍せずそのまま、
        %APPDATA%\.minecraft\resourcepacks\ フォルダへ入れてください。
        ゲーム内「設定 > リソースパック」で本パックを有効側へ移動してください。
  ```
