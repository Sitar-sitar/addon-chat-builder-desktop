現在の Java 版生成は 3 パターン（shaped レシピ / 定期チャット通知 / lang 上書き）に限定されており、

**capability カタログを単一の真実源**とする構造へ改修し、対応パターンを段階的に拡張する。

## 1. 目的・背景

v0.2.0 の実運用初日に、次の不具合構造が確認された（2026-07-19 生成物調査）：

- ユーザーの「夜間に暗視効果を定期付与」という要望に対し、生成された `main.mcfunction` は
  `tellraw`（チャット通知）と `schedule`（自己再予約）のみで、`effect give` が存在しなかった。
- 根本原因は 3 点：
  1. **生成能力が固定テンプレート 1 種**（tellraw 通知）しかなく、`AddonSpec.script` に
     「何をするか」を表すモデルが無い（`message` に意図が押し込まれる）。
  2. **チャット整理プロンプトが能力の限界を宣言していない**ため、LLM が未対応要望を受理し、
     意図を `message` と `description` に流し込んで「対応できたことにする」。
  3. **description が LLM の自由記述のまま pack.mcmeta へ転記**され、実挙動と乖離した
     自己申告を持つ成果物が正常ビルドされる。

本設計の狙い：

- データパック・リソースパックで実現できるパターンを体系的に調査し（§4）、
  「テンプレート化して安全に生成できるもの」を選定して対応範囲を大きく広げる。
- 対応能力（capability）の定義・リリース状態・版要件・文言を**カタログ 1 箇所**へ集約し、
  検証・プロンプト・UI・生成器がすべて同じ capability 判定を経由する構造にする。
  「LLM が知っている能力」「UI が出す能力」「生成器が作れる能力」の乖離を構造的に無くす。
- **未対応・未解決の要望が残っている間は生成自体を止める**（build gate。§5.4）。
  「要望の一部だけ実現して成功扱いにする」経路を仕様として塞ぐ。

## 2. スコープ

### 2.1 IN

- **capability カタログ基盤**（新規 `pattern-catalog.ts`）：原子的 capability（トリガー・アクション・
  レシピ型・loot・リソースパックパターン）の宣言、リリース済みフェーズ定数
  `RELEASED_JAVA_PHASE`、版要件判定、プロンプト断片・表示文言の単一真実源化
- **Java 専用スクリプトモデル `javaScript` の新設**：トリガー（interval → Phase 3 でイベント追加）＋
  発火条件（常時 / 昼 / 夜 / 雨 / 雷雨）＋アクション列（メッセージ / エフェクト付与 /
  タイトル・アクションバー / サウンド / 時刻・天候固定）。
  **既存 Bedrock の `script`（`event`/`message`）は無変更で凍結**（JAVA-MULTI-REV-P1-02）
- **spec の edition 別完全分離**：`AddonSpec` を `BedrockAddonSpec | JavaAddonSpec` の
  discriminated union にし、`unsupportedRequests` 等の新設フィールドは Java 側のみに置く。
  Codex 送信は `toLegacyBedrockCodexSpec()` で v0.2.0 のキー集合・順序へ明示射影し、
  Bedrock の build gate 条件・`BuildResult.description` の値も本設計で固定（JAVA-MULTI-REREV-P1-01）
- **build gate**：`validateSpec` 空 ∧ `unresolvedQuestions` 空 ∧ `unsupportedRequests` 空 のときだけ
  生成可能。client 表示と server `/api/build` の双方で同一判定（JAVA-MULTI-REV-P1-03）
- **レシピ種類の拡張**：shapeless・かまど系 4 種・stonecutting・smithing_transform
- **イベント駆動スクリプト**：食べた時 / 設置時 / 討伐時（advancement 方式・JSON 全文定義）、
  採掘時 / 死亡時（scoreboard 方式）
- **ルートテーブル基本形**：ブロックドロップの差し替え
- **リソースパック拡張（JSONのみ）**：`resourcepack.pattern` を新設し、lang（既存）に加えて
  アイテムモデルの差し替え（itemModelSwap、1.21.4+）を追加
- **最新バージョン対応**：`JAVA_VERSIONS` へ **26.2（Data Pack 107.1 / Resource Pack 88.0）** を追加し、
  1.21.9 以降の `pack.mcmeta` 新形式（`min_format`/`max_format`、minor 付き配列）へ対応。
  `PackFormat` 型を配列対応にし、版差 capability 列（§5.6）で「行追加だけで新版を吸収」を実現
- **description の決定論化と API 契約**：`describePack` の文法表を定義し、pack.mcmeta / README /
  ビルド結果（`BuildResult.description` 新設）の 3 表示を同一文字列に固定
- **版依存検証のデータフロー**：`validateSpec(spec, capabilities)` 契約・`/api/config` の capability 配信・
  server 側の独立再検証（JAVA-MULTI-REV-P2-01）
- **ID・objective 命名契約**：resource location 正規表現の一元化、パス用途 ID の制約、
  scoreboard objective の決定論的短縮命名（JAVA-MULTI-REV-P2-04）
- 上記に伴う `specChecks` / OpenAI スキーマ（edition 別分離）/ UI の追従と、
  対応版×capability の生成マトリクステスト（JAVA-MULTI-REV-P2-05）

### 2.2 OUT（今回触らない・既存挙動を保つ）

- Bedrock 側の生成仕様（`script` 型・Codex プロンプト・スキーマ・白リスト・.mcpack 出力）は
  **完全に現行維持**。Bedrock の chat スキーマも現行のまま分離保持する（§5.2）
- **カスタム風アイテム（名前・lore・エンチャント等のコンポーネント付与）**：
  必要な spec・版別 JSON・検証を本設計内で閉じられないため Phase 4 から除外し、
  **別の機能設計書として分離**する（JAVA-MULTI-REV-P1-05 の解消条件に従う）。
  これに伴い `componentSyntax` 列も本設計から削除（YAGNI。別設計で再定義）
- テクスチャ（PNG）・サウンド（OGG）等バイナリ資産の同梱（v0.2.0 設計 §4.8 の将来枠のまま）
- ワールド生成・ディメンション・ストラクチャ（.nbt）・データ駆動エンチャント
- モブ AI 変更・新規モブ・GUI 追加など、バニラのデータパックで原理的に不可能なもの
- WPF/WebView2 ラッパー・desktop 認証・ポート・DB
- `JAVA_TARGET_VERSION` の既定値変更（現行 `1.21.7` を維持。26.2 は表に追加するが既定にはしない）

## 3. 全体方針

1. **決定論的テンプレート方式を堅持する**（v0.2.0 レビュー JAVA-REV-P1-02 の設計判断を継承）。
   パターンを増やす手段は「LLM に生成させる」ではなく「固定テンプレートを増やす」。
   生成されるコマンド・JSON はすべて本設計 §5.3 に全文を定義し、ユニットテストで完全一致を固定する。
2. **capability カタログ＝可用性と文言の単一真実源**（JAVA-MULTI-REV-P1-01）。
   カタログは「どの capability が・どのフェーズで・どの版で使えるか」と表示文言だけを持つ。
   検証規則の正本は `spec.ts`、生成テンプレートの正本は `java-generator.ts` に置くが、
   両者は **`Record<JavaCapabilityId, …>` の完全マップ**で実装し、TS の網羅性チェックと
   ユニットテストで「カタログにあるのに検証/生成が無い」「無いのに生成できる」を検出する。
   完全マップの検査対象は**そのリリース時点でコードに存在する capability 集合**であり、
   将来 Phase の ID を未実装 stub で埋めない（§5.1 の段階実装契約。JAVA-MULTI-REREV-P1-02）。
3. **できないことは「できない」と言い、できるまで生成させない**。
   未対応要望（`unsupportedRequests`）・未解決質問（`unresolvedQuestions`）が残る間は
   build gate が生成を止める。要望の切り捨て（actions の黙示的な削減等）は行わない。
4. **段階リリース**（§6）。フェーズごとに実ゲーム受入を通してから次フェーズへ進む。
   リリース済みフェーズは `RELEASED_JAVA_PHASE` 定数 1 箇所で管理し、ロールバックは
   この値を下げるだけで成立させる。**Phase 1 の着手前提として v0.2.0 の実ゲーム受入を先に閉じる**
   （JAVA-MULTI-REV-P1-07）。
5. 迷ったら単純な方：聞き取り項目は最小限（既定値を厚く）、ID 実在確認はしない
   （形式検証＋README 注意書き。v0.2.0 確定事項 6 を継承）。

## 4. 調査結果：データパック・リソースパックのパターン全体像と採否

2026-07-19 調査。対象は本ツールがサポートする Java 1.21〜1.21.7 および最新安定版 26.2
（Minecraft Java は 1.21.9 以降、年次形式のバージョン番号 26.x へ移行している。
ディレクトリ名は 1.21 で単数形へ統一済みのため、全対応版で単数形のみ扱う）。
版差の公式根拠は設計レビュー（JAVA-MULTI-REV-P1-04 / P2-02）が照合した
minecraft.net の 1.20.5 / 1.21.4 / 1.21.9 / 26.1 / 26.2 リリースノートに依る。
残る書式確認は §13 の実装時検証ゲートで版別 fixture へ固定する。

**判定凡例**: ◎=テンプレート化容易（採用） / ○=テンプレート化可能・条件付き（採用） /
△=複雑・将来検討 / ×=対象外（理由を明記）

### 4.1 データパックのコンテンツ種別と採否

| ディレクトリ（`data/<ns>/…`） | 内容 | 判定 | 方針 |
|---|---|---|---|
| `recipe/` | クラフト・精錬等のレシピ | ◎ | Phase 2 で種類を拡張（§4.2 R群） |
| `function/` + `tags/function/` | mcfunction・load/tick タグ | ◎ | 固定コマンドテンプレートのみ生成（自由記述コマンドは受けない）。tick.json は引き続き不使用 |
| `predicate/` | 条件判定（時間帯・天候等） | ◎ | 発火条件（昼/夜/雨/雷雨）の実現手段として採用（§5.3.2） |
| `advancement/` | 進捗（イベント検知・報酬にも使える） | ○ | Phase 3 のイベント駆動トリガーとして採用（非表示の技術用進捗。JSON 全文は §5.3.4）。トースト表示用のカスタム進捗は△（将来） |
| `loot_table/` | ドロップ・チェスト内容 | ○ | Phase 4 で「ブロックドロップ差し替え」のみ採用。モブドロップの「追加」はバニラ表の複製が必要で保守困難のため△ |
| `item_modifier/` | アイテム加工関数 | △ | 採用パターンに必要になるまで見送り（YAGNI） |
| `tags/`（block/item/entity_type 等） | タグ追加 | △ | 単体では用途が薄い。必要になったパターンの内部部品としてのみ使用 |
| `enchantment/` | データ駆動エンチャント（1.21+） | △ | スキーマが大きく版差も激しい。将来検討 |
| `damage_type/` `chat_type/` `trim_*/` `jukebox_song/` 等 | 各種レジストリ | × | 利用頻度が低くテンプレート数が発散する。要望が出た時点で個別検討 |
| `worldgen/**`・`dimension*/` | ワールド生成・ディメンション | × | 影響が世界全体に及び失敗時の被害が大きい。チャットで安全に聞き取れる粒度を超える |
| `structure/`（.nbt） | ストラクチャ | × | バイナリ NBT が必要（テキスト生成不可） |
| `pack.mcmeta` の `filter` | 下位パックのファイル除去（バニラレシピ無効化等） | △ | 「レシピ削除」パターンの実現手段候補として将来検討 |

### 5.1 capability カタログ（新規 apps/web/src/lib/pattern-catalog.ts）

**原子的 capability** で表現する（JAVA-MULTI-REV-P1-01）。トリガーとアクションを分離するため、

```ts
  | "script.trigger.killEntity"   // Phase 3
  | "script.trigger.mineBlock"    // Phase 3

  promptLine: string;                     // system プロンプトの対応範囲へ機械挿入する 1 行（対応条件・聞き取る項目）
  starterPrompt?: string;                 // UI スターター例に出す代表入力文（代表 capability のみ設定。

};

export const JAVA_CAPABILITIES: readonly JavaCapability[];
// ★段階実装の契約（JAVA-MULTI-REREV-P1-02）：本設計書には全 Phase の ID を計画として記載するが、
//   コード上の JavaCapabilityId 型・JAVA_CAPABILITIES・validator/generator マップへは
//   「各 Phase の実装時に、ID と handler を同時に追加」する。将来 Phase の ID を
//   未実装 stub で登録してはならない。完全マップの網羅検査は常に
//   「その時点でコードに存在する ID 集合」に対して成立させる（TS 網羅性チェック＝コンパイルエラー）。
//   RELEASED_JAVA_PHASE は「実装済みだが未リリース」を隠すロールバック用ゲートとして併用する。

// リリース済みフェーズの唯一の正本（ビルド時定数）。ロールバックはこの値を下げるだけ。
export const RELEASED_JAVA_PHASE: 0 | 1 | 2 | 3 | 4 | 5;

export function enabledJavaCapabilities(
  rule: JavaVersionRule,
  releasedPhase: number = RELEASED_JAVA_PHASE
): JavaCapability[];
// phase <= releasedPhase かつ requires の全列が true のものだけを返す

export function capabilitiesForSpec(spec: AddonSpec): JavaCapabilityId[];
// spec が実際に使用している capability を列挙する（trigger 1 + condition 依存なし + action×n + recipe型 等）。
// validateSpec / プロンプト生成 / UI / java-generator はすべてこの関数と
// enabledJavaCapabilities の差集合で「使えない capability の使用」を判定する
```

- **責務の線引き**：カタログの正本は「可用性（phase・版要件）と文言」のみ。
  検証規則の正本は `spec.ts`、生成テンプレートの正本は `java-generator.ts`。
  v0.7.1 の現行実装では `Record<JavaCapabilityId, true>` による ID 登録漏れの型検知と、
  版×capability テストを併用する。真偽値表は実 handler の網羅を保証しないため、
  生成内容の網羅性はテスト側で担保する（後続レビューの残課題は別記録）。
- **発火条件（always/day/night/rain/thunder）は capability にしない**：全条件が Phase 1 で同時に
  リリースされ版差もないため、`javaScript.condition` の列挙値として `spec.ts` が検証する。
- **網羅テスト**（JAVA-MULTI-REV-P1-01 解消条件）：
  - 全 `JavaCapabilityId` について validator マップ・generator マップ・`promptLine` が存在する
  - `releasedPhase` を 1〜5 に振ったとき、未リリース capability が
    validateSpec / プロンプト / UI スターター / generator のすべてで拒否される

**ID・命名の検証契約**（JAVA-MULTI-REV-P2-04。正規表現は `spec.ts` に 1 箇所で定義）：

```ts
export const MC_ID = /^minecraft:[a-z0-9_]+$/;          // アイテム・ブロック・エフェクト・エンティティ ID
export const MC_SOUND_ID = /^minecraft:[a-z0-9._]+$/;   // サウンド ID（ドットを許容）
export const MC_PATH_ID = MC_ID;                        // ファイルパスへ使う ID。単一セグメントのみ
// （slash・ドット・空文字を許容しないため、`minecraft:` 除去後の path 部を
//   そのままファイル名 1 セグメントとして使ってもディレクトリトラバーサルが構造的に発生しない）
```

- `namespace` は既存 `^[a-z][a-z0-9_]*$` に **長さ 1〜32** を追加する。
- **scoreboard objective 名**：namespace から直接作らず、
  `"acb_" + sha1(<namespace> + ":" + <trigger> + ":" + <triggerBlockId>).slice(0, 11)`
  （合計 15 文字・`[a-z0-9_]`）とする。全対応版の objective 名制約（16 文字以内）に収まる。
  短縮 hash（44bit）である以上、衝突の可能性は**確率的に十分低いがゼロではない**
  （JAVA-MULTI-REREV-P2-01）。そのため**生成器は同一パック内で生成した objective 名の集合を検査し、
  重複を検出したら build を明示エラーにする**（自動リネーム・salt 再計算は行わない＝単純側に倒す）。

**`validateSpec(spec, capabilities, rule)` / `specChecks(spec, capabilities, rule)` の契約**
（JAVA-MULTI-REV-P2-01。不変条件 `specChecks ⇔ validateSpec` は引数を揃えたうえで維持）：

- **Bedrock（`BedrockAddonSpec`）の検証は v0.2.0 の規則から一切変更しない**（第 2・第 3 引数は無視する）。
  以降の規則はすべて `JavaAddonSpec` にのみ適用される。
- 第 2 引数 `capabilities: JavaCapabilityId[]` は「現在の対象版×リリース済みフェーズで有効な
  capability の集合」。client は `/api/config` の応答から、server は `resolveJavaVersion()` ＋
  `enabledJavaCapabilities()` から**独立に**取得する（§5.5・§5.7）。
- 第 3 引数 `rule?: JavaVersionRule` は版依存検証（cooking の `resultCount` 版別制約 ＝
  `rule.cookingResultCount`。下記 recipe 検証で使用）に渡す。client は `/api/config` の
  `javaVersionRule` から、server は `resolveJavaVersion().rule` から**独立に**取得する
  （§5.5・§5.7）。Bedrock では未使用。省略時は cooking の版別 resultCount 検証をスキップする
  （他の検証は不変）。
- `capabilitiesForSpec(spec)` の結果が `capabilities` の部分集合でなければエラー
  （「未リリース or 版要件未達の capability を使う spec」を全経路で拒否）。
- Java × script の追加検証：
  - actions 1〜3 件。type 別必須（message/title/actionbar → text 非空、effect → effectId MC_ID ＋
    effectSeconds 範囲 ＋ amplifier 0〜9、playsound → soundId MC_SOUND_ID、setTime/setWeather → 列挙値非空）
  - trigger=interval：intervalSeconds 5〜3600、effect は `effectSeconds >= intervalSeconds + 15`
  - trigger 別必須：consumeItem → triggerItemId、placedBlock/mineBlock → triggerBlockId、
    killEntity → triggerEntityId（すべて MC_ID。mineBlock は MC_PATH_ID）
- recipe の追加検証（JAVA-MULTI-REV-P2-02）：
  - recipeType 別必須：shapeless → ingredients 1〜9、cooking → inputItem ＋ cookingXp 0〜100 ＋
    cookingSeconds 1〜600、stonecutting → inputItem、smithing → template/base/addition
  - `resultCount`：shaped/shapeless/stonecutting は 1〜64（全版）。
    **cooking は `rule.cookingResultCount` が false の版では 1 のみ**（2 以上はエラー文言で拒否）、
    true の版（26.1+）では 1〜64。**smithing は全版 1 固定**（UI の個数行も 1 固定表示）
- loot：targetBlockId は MC_PATH_ID、dropItemId は MC_ID、dropCount 1〜64
- resourcepack：pattern=lang は現行 langEntries 検証。pattern=itemModelSwap は
  targetItem・sourceItem とも MC_PATH_ID 必須、targetItem ≠ sourceItem

### 5.3 生成テンプレート（apps/web/src/lib/java-generator.ts）

生成物はすべて以下の固定形。`<>` 部分のみ spec から埋める。JSON は `JSON.stringify` 構築（踏襲）。
生成器は `Record<JavaCapabilityId, GeneratorFn>` の完全マップで実装する（§5.1）。

#### 5.3.1 pack.mcmeta と description の決定論化（全パターン共通・Phase 1）

**pack.mcmeta（`mcmetaFormat` 世代で分岐。JAVA-MULTI-REV-P1-04）**

```jsonc
// legacy（1.21〜1.21.7）: PackFormat は number
{"pack":{"pack_format":<number>,"description":"<describePack の出力>"}}

// minMax（1.21.9+ / 26.x）: PackFormat は [major, minor] の整数 2 要素配列
{"pack":{"description":"<describePack の出力>","min_format":[<major>,<minor>],"max_format":[<major>,<minor>]}}
```

- min_format / max_format はともに対象版の値を設定する（範囲指定はしない。単純側に倒す）。
- 26.2 の実値：datapack `[107,1]`、resourcepack `[88,0]`（§5.6 の表に固定済み）。

**describePack（3 表示統一。JAVA-MULTI-REV-P2-03）**

`describePack(spec, version): string` を pure module として実装し、
**pack.mcmeta の description・README の概要行・`BuildResult.description`（§5.7）** の 3 出力を
同一文字列にする（integration test で固定）。文法表：

| kind | 文型 |
|---|---|
| script | `{条件句}{トリガー句}、{アクション句（「・」区切り）}するデータパック（Java {版}）` |
| recipe | `{設備句}で{result 表示}を作るレシピ（Java {版}）` |
| loot | `{targetBlock 表示}のドロップを{dropItem 表示}×{dropCount}へ差し替えるデータパック（Java {版}）` |
| resourcepack(lang) | `表示名を{n}件変更するリソースパック（Java {版}）` |
| resourcepack(itemModelSwap) | `{targetItem 表示}の見た目を{sourceItem 表示}へ差し替えるリソースパック（Java {版}）` |

- トリガー句：interval → `{intervalSeconds}秒ごとに` / consumeItem → `{item 表示}を食べたときに` /
  placedBlock → `{block 表示}を設置したときに` / killEntity → `{entity 表示}を倒したときに` /
  mineBlock → `{block 表示}を掘ったときに` / death → `死亡したときに`
- 条件句：always → 空 / day・night → `昼/夜の間だけ` / rain → `雨の間だけ` / thunder → `雷雨の間だけ`
- アクション句：message → `チャット通知` / effect → `{effect 表示}付与（強さ{amplifier+1}・{effectSeconds}秒）` /
  title → `タイトル表示` / actionbar → `アクションバー表示` / playsound → `サウンド再生` /
  setTime → `時刻を{timeValue}に固定` / setWeather → `天候を{weatherValue}に固定`
- 設備句：shaped/shapeless → `作業台` / cooking 4 型 → `かまど/溶鉱炉/燻製器/焚き火` /
  stonecutting → `石切台` / smithing → `鍛冶台`
- **ID の表示規則**：`minecraft:` を除いた path をそのまま表示（例 `night_vision`）。
  30 文字超は先頭 30 文字＋`…`。**ユーザー自由記述（message text 等）は description に含めない**
  （決定論と長さ上限の担保。text は tellraw/title 本体にのみ載る）。
- LLM の `spec.description` は会話・タイトル用途のみに残し、成果物ファイルへは載せない。

#### 5.3.2 発火条件（predicate。Phase 1）

condition が `always` 以外のとき、`data/<ns>/predicate/<condition>.json` を生成する。

```jsonc
// night（12542〜23459 ＝ ベッド使用可能帯。確定事項 4）
//   timeCheckClock=false の版（1.21 / 1.21.4 / 1.21.5 / 1.21.7）
{"condition":"minecraft:time_check","period":24000,"value":{"min":12542,"max":23459}}
//   timeCheckClock=true の版（26.2）：World Clock ID を明示する（JAVA-MULTI-REREV-P1-03）
{"condition":"minecraft:time_check","clock":"minecraft:overworld","period":24000,"value":{"min":12542,"max":23459}}

// day（夜の補集合。any_of の terms 内の各 time_check にも同じ版分岐を適用）
{"condition":"minecraft:any_of","terms":[
  {"condition":"minecraft:time_check","period":24000,"value":{"min":0,"max":12541}},
  {"condition":"minecraft:time_check","period":24000,"value":{"min":23460,"max":23999}}]}

// rain（全版共通）
{"condition":"minecraft:weather_check","raining":true}
// thunder（全版共通）
{"condition":"minecraft:weather_check","thundering":true}
```

- **time_check の出力形は本設計で確定**する：`timeCheckClock: false` の版は `period`/`value` のみ、
  `true` の版（26.2）は `"clock":"minecraft:overworld"` を追加した形を出力する
  （`timeCheckClock` 列はこの分岐で実際に使用される＝「版差は表の列で吸収」の契約を満たす）。
- **対応 5 版（1.21 / 1.21.4 / 1.21.5 / 1.21.7 / 26.2）の day・night predicate 全文 fixture** を
  受入条件に置く（§11）。公式仕様・実機確認が本形と異なる場合は、
  **実装を始める前に本設計書を改訂して再々レビュー**する（実装側で分岐や形を決める運用はしない）。

#### 5.3.3 アクション行の展開規則（interval / event 共通の正本表。JAVA-MULTI-REV-P1-06）

`<cond>` は condition ≠ always のとき `execute if predicate <ns>:<condition> run ` を前置。
condition = always のときは前置なし。**この規則は interval・event の全アクション行に一律適用**する。

| action | trigger=interval（対象＝全員） | trigger=イベント系（対象＝発火した @s） |
|---|---|---|
| message | `<cond>tellraw @a {"text":"<text>"}` | `<cond>tellraw @s {"text":"<text>"}` |
| effect | `<cond>execute as @a run effect give @s <effectId> <effectSeconds> <amplifier> true` | `<cond>effect give @s <effectId> <effectSeconds> <amplifier> true` |
| title | `<cond>title @a title {"text":"<text>"}` | `<cond>title @s title {"text":"<text>"}` |
| actionbar | `<cond>title @a actionbar {"text":"<text>"}` | `<cond>title @s actionbar {"text":"<text>"}` |
| playsound | `<cond>execute as @a at @s run playsound <soundId> master @s ~ ~ ~ 1 1` | `<cond>execute at @s run playsound <soundId> master @s ~ ~ ~ 1 1` |
| setTime | `<cond>time set <timeValue>` | `<cond>time set <timeValue>`（グローバル。1 回のみ実行） |
| setWeather | `<cond>weather <weatherValue>` | `<cond>weather <weatherValue>`（同上） |

- effect は `true`（パーティクル非表示）固定。interval では `effectSeconds >= interval + 15` の検証
  （§5.2）により暗視の残り 10 秒未満での画面点滅を構造的に回避する。
- `tick.json` 不使用の方針は全パターンで継続（JAVA-REV-P1-03 継承）。

**生成 handler の合成契約（JAVA-MULTI-REREV-P1-02）**

script の生成は「trigger handler 1 件＋action handler 1〜3 件＋condition wrapper」の合成で行う。
入出力を型で固定する：

```ts
type TriggerHarness = {
  files: GeneratedPackFile[];         // load.mcfunction / main.mcfunction / on_event.mcfunction /
                                      // advancement JSON など、trigger handler が所有する骨格ファイル
  actionInsertFile: string;           // action 行を挿入するファイルの path（interval → main / イベント → on_event）
  actionColumn: "interval" | "event"; // 上表のどちらの列の形で action 行を作るか
  epilogue: string[];                 // 挿入後に必ず末尾へ置く行（schedule 再予約 / advancement revoke / スコアリセット）
};
type ActionFragment = string;         // action handler が返す 1 コマンド行（<cond> 前置前）
```

- **合成順**：trigger handler が骨格を生成 → action handler を **`actions` の宣言順**に fragment 化 →
  condition wrapper が各 fragment へ `<cond>` を前置 → `actionInsertFile` へ宣言順のまま挿入 →
  `epilogue` を**無条件で**末尾に付加（§5.3.4 の「revoke/リセットは条件不成立でも常に実行」を
  この合成規則で担保する）。
- **ファイル所有権**：predicate JSON（§5.3.2）は condition wrapper が、それ以外の script 系ファイルは
  trigger handler が所有する。合成結果で生成ファイルの path が重複した場合は
  **build を明示エラー**にする（黙って上書き・マージしない）。

**script × interval（Phase 1。既存 DP-S1 を包含）**

```
# data/<ns>/function/load.mcfunction
schedule function <ns>:main <interval>s replace

# data/<ns>/function/main.mcfunction
<アクション行を宣言順に展開（上表の interval 列）>
schedule function <ns>:main <interval>s replace

# data/minecraft/tags/function/load.json（既存踏襲）
{"values":["<ns>:load"]}
```

#### 5.3.4 script × イベント駆動（Phase 3）

**advancement 方式（consumeItem / placedBlock / killEntity）— JSON 全文（JAVA-MULTI-REV-P1-06）**

`data/<ns>/advancement/on_event.json`（`display` キーなし＝画面に出ない技術用進捗）：

```jsonc
// consumeItem
{"criteria":{"trigger":{"trigger":"minecraft:consume_item",
  "conditions":{"item":{"items":["<triggerItemId>"]}}}},
 "rewards":{"function":"<ns>:on_event"}}

// placedBlock
{"criteria":{"trigger":{"trigger":"minecraft:placed_block",
  "conditions":{"location":[{"condition":"minecraft:location_check",
    "predicate":{"block":{"blocks":["<triggerBlockId>"]}}}]}}},
 "rewards":{"function":"<ns>:on_event"}}

// killEntity
{"criteria":{"trigger":{"trigger":"minecraft:player_killed_entity",
  "conditions":{"entity":[{"condition":"minecraft:entity_properties","entity":"this",
    "predicate":{"type":"<triggerEntityId>"}}]}}},
 "rewards":{"function":"<ns>:on_event"}}
```

`data/<ns>/function/on_event.mcfunction`：

```
<アクション行を宣言順に展開（§5.3.3 の イベント列。各行に <cond> を個別適用）>
advancement revoke @s only <ns>:on_event
```

**condition の状態遷移（確定仕様）**：advancement の criteria には condition を組み込まない。
発火条件は `on_event.mcfunction` 内で**各アクション行にのみ** predicate guard として適用し、
**末尾の `advancement revoke` は条件不成立でも常に実行**する。
→「夜にリンゴを食べたら回復」で、昼に食べた場合は何も起きず達成状態も残らない
（昼の達成が夜に遅延発火する経路が存在しない）。

**scoreboard 方式（mineBlock / death）**

```
# data/<ns>/function/load.mcfunction（<obj> は §5.2 の決定論 objective 名）
scoreboard objectives add <obj> <criteria>
schedule function <ns>:main 1s replace
#   criteria: mineBlock → minecraft.mined:minecraft.<triggerBlockId の path> / death → deathCount

# data/<ns>/function/main.mcfunction（1 秒周期の検知ループ）
execute as @a if score @s <obj> matches 1.. run function <ns>:on_event
schedule function <ns>:main 1s replace

# data/<ns>/function/on_event.mcfunction
<アクション行を宣言順に展開（§5.3.3 の イベント列。各行に <cond> を個別適用）>
scoreboard players set @s <obj> 0
```

- スコアのリセットも revoke と同様、**条件不成立でも常に実行**する（遅延発火を残さない）。
- 検知遅延最大 1 秒は仕様として README に明記する（tick.json 不使用とのトレードオフ）。

### 5.4 OpenAI 連携（apps/web/src/lib/openai.ts）

- **chat スキーマを edition 別に分離**（JAVA-MULTI-REV-P1-02）：
  - `bedrockChatResponseSchema`：**現行スキーマを 1 バイトも変えない**（`script.event` に
    `itemUse`/`blockBreak`/`interval` を保持）。normalize も現行維持で、戻り値は
    `BedrockAddonSpec`（`unsupportedRequests` は生成しない＝フィールド自体が存在しない）。
    Codex へ渡す user input の `text` は v0.2.0 と完全一致（fixture 全文一致の回帰テストで固定。§10）。
  - `javaChatResponseSchema`：§5.2 の `javaScript`（`actions` は **`maxItems: 3`** をスキーマで宣言）・
    `recipe`（recipeType ほか）・`loot`・`resourcepack.pattern`・`unsupportedRequests` を含む新スキーマ。
    strict 全項目必須・未使用は安全な既定値、の現行方式踏襲。
- **Java 用 system プロンプトの「対応範囲」節をカタログから機械生成**：
  `enabledJavaCapabilities(rule)` の `promptLine` を列挙して挿入する。手書きの対応範囲列挙は削除。
- **拒否規則**（JAVA-MULTI-REV-P1-03）：
  - 「カタログ一覧に無い挙動（例：新モブ追加、ワールド生成変更、GUI、モブAI変更、複雑な連鎖条件）は
    『このツールでは未対応』と assistantMessage で明言し、**`unsupportedRequests` に
    `未対応: <要望>` を追加**すること。会話でまだ確認できていない事項は従来どおり
    `unresolvedQuestions` に残すこと」
  - 「未対応の要望を message アクションの文言や description で"実現したことにしない"こと。
    近い対応パターンがあれば suggestedReplies で代替案を提示すること」
  - 「アクションが 4 件以上要望されたら、勝手に削らず『3 件以内に絞る』質問を
    `unresolvedQuestions` に残すこと」
- **normalizeRawSpec は黙示的な切り詰めをしない**：actions が 4 件以上の応答は
  スキーマ違反（maxItems: 3）としてエラー化する。`slice(0, 3)` 相当の処理は実装しない。
  edition ロック・非対象セクション消去（現行）は維持し、edition 別 normalizer に分割する
  （bedrock normalizer は `BedrockAddonSpec` を、java normalizer は `JavaAddonSpec` を返す）。

### 5.5 UI（apps/web/src/app/page.tsx・src/utils/addon-view.ts）

- **`/api/config` 応答の拡張**（JAVA-MULTI-REV-P2-01）：
  `{ javaTargetVersion, javaCapabilities: JavaCapabilityId[], javaVersionRule: JavaVersionRule }` を返す
  （server が `resolveJavaVersion()`＋`enabledJavaCapabilities()` で解決した値）。
  client は `javaCapabilities` を `validateSpec` / `specChecks` / スターター例に、
  `javaVersionRule` を `validateSpec` / `specChecks` の第 3 引数（§5.2）へ渡す。
  版依存検証（cooking の `resultCount` 版別制約）を client でも server と同一に行うため、`rule` も配信する。
  **config 取得前・取得失敗時は Java の chat スターターと生成ボタンを無効化**する。
- **build gate の表示**：生成ボタンの活性条件を
  `validateSpec(spec, caps).length === 0 && unresolvedQuestions.length === 0 && unsupportedRequests.length === 0`
  へ変更。`unsupportedRequests` が残る場合は設計図に「未対応の要望」行（ブロッキング行）を表示する。
- 設計図（BLUEPRINT）行の追加：`script` は「トリガー」「条件」「アクション n 件」行、
  `recipe` は「レシピ種類」行（smithing は個数行を 1 固定表示）、`loot` は「対象ブロック」「ドロップ」行、
  `resourcepack` は「パターン」行。`specChecks` と同値になるよう拡張（不変条件テスト維持）。
- スターター例は catalog entry の `starterPrompt`（§5.1）が設定された capability から
  **決定論的に**生成する（例：「夜だけ暗視をつけたい」「りんごを食べたら回復エフェクト」
  「金インゴットを焼いて作りたい」。`starterPrompt` の無い capability はスターターに出さない）。
- 生成ボタン・出力サフィックス・出力先はパック種別ベースの現行実装のまま
  （`loot` は datapack として `resolvePackType` に追加）。

## 9. リスクと割り切り

- **テンプレート数の増加による保守コスト**：capability の完全マップ＋「生成ファイル全文一致テスト」で
  1 capability 1 テストに封じ込める。カタログ外の分岐をコードに書かない規約とする。
- **コマンド・JSON 書式の版差の見落とし**：§4.4 で把握した差分は `JavaVersionRule` の列で吸収し、
  §10 の版×capability マトリクステストと §13 の実装時検証ゲートで検出する。
  受入で書式差が発覚した場合は表の列追加で対処し、テンプレートに if を散らさない。
- **ID 実在確認をしない割り切りの拡大**：エフェクト・サウンド・エンティティ ID にも同じ方針を
  適用する（形式検証＋README 注意書き）。存在しない ID は該当コマンド行・レシピが無効になるだけで
  パック全体は壊れない（クラッシュしない）ことを受入で確認する。
- **`loot.blockDrop` の完全上書き**：元のドロップ挙動（シルクタッチ等）が失われる。README 明記＋
  チャットで対象ブロックを復唱確認する運用でカバー。
- **build gate の厳格化による UX 低下**：未解決・未対応が 1 件でも残ると生成できなくなる。
  意図した挙動（部分実現より生成不可を選ぶ）だが、チャット側で unresolved を解消しやすいよう
  suggestedReplies の活用をプロンプトに明記する。
- **スキーマ肥大による LLM 整理精度の低下**：edition 別分離により Bedrock 側は不変。
  Java 側は normalizeRawSpec の既定値化で吸収し、聞き取り品質はフェーズごとの手動確認で監視する。
- ロールバック：`RELEASED_JAVA_PHASE` を下げれば、カタログ・検証・プロンプト・UI・生成器が
  一斉に前フェーズ相当へ縮退する（正本が 1 箇所のため）。
