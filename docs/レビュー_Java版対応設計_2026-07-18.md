## レビュー結論

**判定: 着手不可**

既存アプリへエディション属性を追加し、Bedrock の生成フローを保ったまま Java 用出力を分離する方向性は妥当である。しかし、現設計のままでは「テストと受け入れ条件を満たしたのに、Minecraft Java Edition が読み込めない、または意図と異なる高頻度処理を行う zip」を正常終了として出力できる。

実装前に、少なくとも次の P1 3件を設計へ反映し、再レビューする必要がある。

1. 対象を `1.21系` ではなく厳密な Java バージョンへ固定し、データパック／リソースパック別のメタデータ形式をアプリ側で決定する。
2. Java 生成物を JSON 構文だけでなく、種類別の必須ファイル・内容・相互参照まで検証する。推奨は Java 3種類を仕様から決定論的に生成すること。
3. `interval` の時間を仕様に持たせ、毎tickの `tellraw` 実行を避ける。

## 実装前に確定すべき事項

1. 実際に利用する Java Edition の厳密なバージョン（例: `1.21.7`）。「1.21系」だけでは不足する。
2. Java script の「一定間隔」を何秒単位で指定可能にするか。推奨既定値と上限。
3. Javaレシピは、バニラ既存アイテムだけを対象にするか。Mod導入環境の外部namespaceを許すなら、存在確認不能であることを別モードとして明示する必要がある。
4. `javaパックファイル` のzipをGit管理するか、ローカル生成物として除外するか。

## 良い点

- 新規アプリを増やさず `apps/web` に統合する判断は、環境変数・起動経路・UIの二重管理を避けられる。
- editionとpack typeの差を `pack-rules.ts` へ集約する方向は、Bedrock回帰を局所化しやすい。
- Bedrockの認証・WPFライフサイクルを非対象に保つ境界は現行コードと整合している。WPFは `.env` の任意キーをWebプロセスへ渡すため、`DEFAULT_OUTPUT_DIR_JAVA` とJava設定の追加だけならWPF変更は不要である（`src/AddonChatBuilder.Desktop/Services/AppSettingsService.cs:83-126`, `WebAppProcessService.cs:67-73`）。
- エディション切替時に会話と仕様をリセットするUX方針、Java専用の出力suffix／出力先、Bedrock回帰テストを明記した点は妥当である。

## 推奨修正順

1. Javaの厳密な対象バージョンと、データ／リソース別pack metadataを確定する。
2. Java生成を決定論的テンプレートへ変更し、種類別の必須ファイル・内容検証を設計する。
3. `intervalSeconds` とschedule方式を設計する。
4. edition lock、非同期切替競合、UIへの対象バージョン供給経路を設計する。
5. テスト表と受け入れ条件を更新し、3種類の実ゲーム確認を必須化する。
6. ブランチ、実装ログ、索引、README、親台帳、Obsidian、zipのGit方針を完了条件へ追加する。
7. P1/P2を設計書へ反映後、再レビューで着手可否を判定する。
