## レビュー結論

**判定: 条件付き着手可**

M1 の未使用 key 記号拒否は現行 `validateJavaRecipe` の不足を正しく補い、M2 の依存方向も実行時循環を作らない。M3 のタイムアウト追加、M4 の no-op 削除、M5 の利用者向け注意追加も目的は妥当であり、現時点で P1 ブロッカーはない。

ただし、設計が主張する「無効 ZIP を生成しない」「タイムアウトが効く」「capability の実装漏れを検知する」「衝突条件を正しく伝える」を受け入れ時に証明できない、または現行コードの実態と一致しない P2 が5件ある。以下を修正設計書へ反映し、実装完了条件として閉じることを条件に着手可とする。

## 良い点

- M1 は既存の `symbols` を再利用する最小差分で、空白予約記号の扱いも現行正規化と整合している。
- M2 は capability 判定の正本を生成側と共有し、`pattern-catalog.ts` から `spec.ts` への参照が type-only であることまで確認している。
- M3 を chat / Codex 共通 helper へ入れるため、重複実装を増やさない。
- M1 を必須不具合、M2〜M5 を保守対応として分け、v0.7.0 の残実ゲーム受入を別スコープに保った境界は明確である。
- ロールバックを1コミットにまとめ、生成物の意図した差分を README に限定する方針は、小規模運用に適している。

## 推奨修正順

1. M4 の網羅性保証を「実 handler map」か「coverage 表＋内容テスト」のどちらにするか確定する。
2. M5 の警告対象を script 限定または pack kind 別へ修正する。
3. M1 の `buildPack` 境界テストと M3 の timeout / fallback テストを追加する。
4. v0.7.1 の実装ログ開始・締め、設計書・索引の状態同期を完了条件へ追加する。
5. P2 反映後、対象修正設計書の変更履歴へ指摘 ID と反映箇所を記録する。
