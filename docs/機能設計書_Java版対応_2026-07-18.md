# 設計書：Java版対応（データパック＋リソースパックJSONのみ）

現在 Bedrock（統合版）専用の Addon Chat Builder（`apps/web`）に、Minecraft Java Edition 向けの
データパック（.zip）とリソースパック（JSONのみの .zip）の生成機能を追加する。
テクスチャ（PNG）同梱は今回実装しないが、将来追加できる構造にしておく。

- **作成日**: 2026-07-18
- **対象アプリ**: addon-chat-builder-desktop（`apps/web`。WPF/WebView2 ラッパーは変更しない）
- **種別**: 機能設計書
- **対象バージョン**: apps/web v0.2.0（採番予定。現行 package.json は 0.1.0）
- **前提コミット**: 8617e49（ブランチ `codex-migrate-web-monorepo`）
- **ステータス**: 着手前（正本）
- **関連**: [design-consolidation.md](design-consolidation.md)（正本を apps/web へ一本化する設計・着手前）/ [design-ui-optimization.md](design-ui-optimization.md)（現行UIの正本・実装済み）

---

## 1. 目的・背景

- 現状は Bedrock 用アドオン（.mcpack）のみ生成できる。Java 版でも使いたい。
- 事前判断（2026-07-18）：新規アプリは作らず**既存アプリへの機能追加**とする。
  理由：チャットUI・OpenAI/Codex連携・生成ファイル検証・zip化パイプラインの大半が
  エディション非依存で再利用でき、新規アプリだと .env / ランチャー / ポートの二重管理になる（YAGNI違反）。
- **実装先の確定（2026-07-18）**：ユーザーが日常利用している画面（草原テーマ＋設計図UI）は
  本リポジトリ `apps/web`（ブランチ `codex-migrate-web-monorepo`）であることをスクリーンショットで
  確認した。単体の旧 `addon-chat-builder` ではなく **`apps/web` を実装先**とする
  （[design-consolidation.md](design-consolidation.md) の一本化方針とも整合）。
- Java 対応範囲はユーザー決定済み：
  - **データパック**（レシピ・mcfunction）
  - **リソースパック（JSONのみ）**。テクスチャ（PNG）は将来対応だが、仕組みは今回考慮しておく。
  - Mod（Forge/Fabric）は対象外。
  - Java 版の出力先は **Java 専用フォルダ**を新設する。

## 2. スコープ

### 2.1 IN

- `AddonSpec` へのエディション概念（`edition: "bedrock" | "java"`）の追加
- Java 用の生成種類：
  - `recipe`（データパック：作業台の shaped レシピ）
  - `script`（データパック：mcfunction。初期は `interval`＝tick 実行のみ）
  - `resourcepack`（リソースパック：lang 上書きによる表示名・文言変更。JSONのみ）
- Java 用 Codex 生成プロンプト・パス白リスト・`pack.mcmeta` 必須化
- 出力形式：Java は `.zip`（データパック / リソースパック）、専用出力先フォルダ
- UI：エディション切替（統合版 / Java版）、エディション別スターター例・設計図行・生成ボタン表示
- 将来のテクスチャ同梱に備えた構造化（§4.7。実装はしない）

### 2.2 OUT（今回触らない・既存挙動を保つ）

- Bedrock 側の生成仕様・検証・.mcpack 出力・既定出力先（現行維持）
- .NET WPF/WebView2 ラッパー（リポジトリ直下 `src/`）と desktop 認証
  （`verifyDesktopRequest` / `DESKTOP_API_TOKEN`）は変更しない
- `/api/chat` の `suggestedReplies` / `recommendedReply` 応答仕様（現行維持）
- 旧単体版 `Minecraft_Addon\addon-chat-builder` は変更しない（撤去は
  [design-consolidation.md](design-consolidation.md) の別判断）
- テクスチャ（PNG）の生成・アップロード・同梱（将来対応）
- Java の Mod（Forge/Fabric）対応
- Java `script` の `itemUse` / `blockBreak` イベント（データパックでの実現が複雑なため将来検討）
- Java でのカスタムアイテム追加（バニラのデータパックでは不可能。§4.1 でエラーにする）
- Bedrock のリソースパック対応
- DB導入・ポート変更

## 3. 全体方針

- **エディションは仕様（AddonSpec）の属性**とし、チャット整理 → 仕様確定 → Codex生成 →
  検証 → zip化 の既存フローはエディション共通のまま維持する。
- エディション・パック種別ごとに異なるもの（パス白リスト・必須ファイル・出力拡張子・
  プロンプト）は**テーブル定義（新規 `pack-rules.ts`）に分離**し、分岐を散らさない。
  これが将来テクスチャ（`textures/*.png`）を白リスト1行の追加で受け入れる布石にもなる。
- Java のパック種別は kind から自動決定する：`recipe` / `script` → データパック、
  `resourcepack` → リソースパック。ユーザーにパック種別を直接選ばせない（1ビルド＝1パック）。
- 現行UIの不変条件 **`specChecks(spec).every(ok) === (validateSpec(spec).length === 0)`**
  （設計図の行完了と生成可否の同値性。`tests/unit/addon-view.test.ts` で担保）は拡張後も維持する。

## 4. 詳細仕様

### 4.1 スペックモデル（apps/web/src/lib/spec.ts）

```ts
export type Edition = "bedrock" | "java";
export type AddonKind = "recipe" | "item" | "script" | "resourcepack";

export type AddonSpec = {
  edition: Edition;          // 追加。既定 "bedrock"
  // ...既存フィールドは維持
  resourcepack?: {           // 追加（Java専用）
    description: string;                       // pack.mcmeta の description
    langEntries: { key: string; value: string }[]; // 例 key: "item.minecraft.diamond_sword"
  };
};
```

- エディション×kind の対応表：

| kind | bedrock | java |
|---|---|---|
| recipe | ○（現行） | ○ データパック |
| item | ○（現行） | ×（バニラデータパックでは不可。validateSpec でエラー文言を返す） |
| script | ○（現行 itemUse/blockBreak/interval） | △ データパック。初期は `interval` のみ |
| resourcepack | ×（今回対象外） | ○ JSONのみ（lang 上書き） |

- `validateSpec` に追加する検証：
  - `edition` が不正 → エラー
  - `java` × `item` → 「Java版ではアイテム追加はデータパックで実現できません（Mod が必要）」
  - `java` × `script` で `interval` 以外 → 「Java版の script は interval のみ対応です」
  - `bedrock` × `resourcepack` → 「リソースパックは Java版のみ対応です」
  - `resourcepack` は `langEntries` 1件以上、key は `[a-z0-9_.]+`
- `specChecks` も同じ内容で拡張する（§3 の不変条件を維持）：
  - `resourcepack` 選択時の行：`langEntries`（「表示名の変更 n件」）等
  - エディション自体は UI 切替で常に確定しているため、ブロッキング行にはしない
    （設計図には §4.6 の情報行として表示）

### 4.2 パックルールのテーブル化（新規 apps/web/src/lib/pack-rules.ts）

エディション×パック種別ごとの規則を1箇所に定義する。

```ts
export type PackType = "mcpack" | "datapack" | "resourcepack";

type PackRule = {
  requiredFile: string;          // manifest.json / pack.mcmeta
  outputSuffix: string;          // ".mcpack" / "-datapack.zip" / "-resourcepack.zip"
  allowedPaths: RegExp[];        // パス白リスト
};
```

| パック種別 | 必須 | 白リスト |
|---|---|---|
| mcpack（bedrock） | manifest.json | 現行維持：`manifest.json`, `README.txt`, `recipes/*.json`, `items/*.json`, `scripts/*.js` |
| datapack（java） | pack.mcmeta | `pack.mcmeta`, `README.txt`, `data/<ns>/recipe/*.json`, `data/<ns>/function/*.mcfunction`, `data/minecraft/tags/function/tick.json`, `data/minecraft/tags/function/load.json` |
| resourcepack（java） | pack.mcmeta | `pack.mcmeta`, `README.txt`, `assets/minecraft/lang/*.json` |

- `<ns>` 部分は実際には `[a-z0-9_]+` で許可する（namespace 完全一致までは要求しない。単純さ優先）。
- `resolvePackType(spec)`: bedrock → `mcpack`、java × recipe/script → `datapack`、java × resourcepack → `resourcepack`。
- **将来のJavaバージョン更新への備え（確定方針）**：対象バージョン→pack_format の対応表は
  持たないが、必要になったら本ファイルに `JAVA_PACK_FORMATS: Record<string, number>` として
  追加する置き場と決めておく（§7 リスク参照）。

### 4.3 生成ファイル検証（apps/web/src/lib/addon-generator.ts）

- `buildMcpack` を `buildPack` に一般化する（戻り値 `{ packPath, files }`。
  `/api/build` の応答フィールドも `mcpackPath` → `packPath` に変更し、page.tsx を追従）。
- 検証をテーブル参照に変更：
  - 必須ファイル：pack 種別の `requiredFile`
  - パス白リスト：pack 種別の `allowedPaths`
  - `.json` の JSON.parse 検証は現行維持（`pack.mcmeta` も JSON なので対象）
  - `.js` の危険パターン検査は現行維持（bedrock のみ発生）
  - **追加**：`.mcfunction` の危険コマンド検査。行頭コマンドで
    `op` / `deop` / `stop` / `ban` / `ban-ip` / `pardon` / `whitelist` / `save-off` を禁止
- zip 化は現行どおり AdmZip。出力ファイル名は `outputName + outputSuffix`
  （例：`my-addon-datapack.zip`）。

### 4.4 出力先（apps/web/src/lib/paths.ts / .env）

- エディション別の既定出力先に変更する：
  - bedrock：現行維持（`DEFAULT_OUTPUT_DIR` → 無指定時 `<cwd>\mcpack`）
  - java：**専用フォルダ**。`DEFAULT_OUTPUT_DIR_JAVA` → 無指定時 `<cwd>\javapack`
- 運用上の配布置き場（.env で指定・確定済み）：
  - bedrock：`D:\my-app2\Minecraft_Addon\mcpackファイル`（現行）
  - java：`D:\my-app2\Minecraft_Addon\javaパックファイル`（新設）
- UI の「既定に戻す」はエディションに応じた既定へ戻す。

### 4.5 OpenAI 連携（apps/web/src/lib/openai.ts）

- チャット整理（refineAddonSpec）：
  - JSON スキーマに `edition` と `resourcepack` を追加（strict スキーマのため全項目必須・
    未使用時は安全な既定値で埋める現行方式を踏襲）。`suggestedReplies` / `recommendedReply`
    の応答は現行のまま維持する。
  - system プロンプトをエディションで分岐。Java 側には
    「対応範囲は recipe / script(interval) / resourcepack(langEntries)」「item は不可、
    代わりに resourcepack の表示名変更か統合版を提案」等を明記する。
  - エディション自体はチャットで変更させない（UI の切替が唯一の入口。混在防止）。
- Codex 生成（generateAddonFilesWithCodex）：
  - system プロンプトをパック種別で分岐。Java 用には
    「対象は Java Edition <JAVA_TARGET_VERSION>」「pack.mcmeta 必須・pack_format は
    対象バージョン準拠」「許可パスは §4.2 の白リスト」「レシピは
    `data/<ns>/recipe/*.json`（`minecraft:crafting_shaped`）」
    「interval script は `data/minecraft/tags/function/tick.json` から
    `<ns>:main` を tick 実行し `say`/`tellraw` を使う」
    「lang は `assets/minecraft/lang/ja_jp.json` と `en_us.json` の両方に同内容を出力」
    「危険コマンド（op/stop/ban 等）を書かない」を指示。
- 環境変数 `JAVA_TARGET_VERSION`（既定 `1.21`・確定済み）を追加し、プロンプトに埋め込む。
  Java 本体の更新で対象を変えたいときは .env の変更だけで済む。

### 4.6 UI（apps/web/src/app/page.tsx・styles.css・src/utils/addon-view.ts）

現行UI（草原テーマ＋「チャット＋ライブ設計図」）の構造は維持し、以下を追加・切替する。

- **トップバー**：`topbar-right`（ハート・リセットの並び）に**エディション切替**
  （ブロック調セグメント：統合版 / Java版）を追加。eyebrow の固定文言
  `MINECRAFT BEDROCK` は選択エディション連動（`MINECRAFT JAVA EDITION`）にする。
- **切替時の挙動**：会話・仕様・生成結果を**リセット**する（エディション混在の仕様を
  作らせない）。作業開始後（hasStarted）は確認ダイアログを出す。
- **スターター例**（chat-hero）をエディション別に切替：
  - Java 例：「新しい武器レシピを作りたい」「アイテムの表示名を変えたい（リソースパック）」
    「一定間隔でチャットにメッセージを流したい」
  - 「新しいアイテムを追加したい」は Java では出さない（データパックでは不可のため）
- **設計図（BLUEPRINT）**：
  - 先頭に**エディション行**（情報行 `info`、値「統合版」/「Java版（1.21）」）を追加
  - Java 時は**パック種別行**（情報行、kind から自動：「データパック」/「リソースパック」）を追加
  - `kindLabel` に `resourcepack` → 「リソースパック」を追加
  - `resourcepack` 選択時のブロッキング行は specChecks 拡張（§4.1）に従い表示
- **生成ボタン（ready-cta と設計図下部の両方）**の文言をパック種別で切替：
  - bedrock：`.mcpack を生成`（現行）
  - java datapack：`データパック(.zip) を生成`
  - java resourcepack：`リソースパック(.zip) を生成`
- **出力先ボックス**：ラベルを「出力先フォルダ（Java版 専用）」等エディション連動にし、
  既定値もエディション別（§4.4）。
- 生成完了メッセージ（「完成！ …を生成しました」）と result-box も同様に切替。

### 4.7 将来のテクスチャ対応への布石（今回は実装しない）

将来「リソースパックにテクスチャ（PNG）を同梱」する際の想定方式を残す：

1. ユーザーが PNG をアップロードする API（例 `/api/assets`）を追加し、
   セッション毎の staging フォルダに保存する。
2. 白リストに `assets/<ns>/textures/**/*.png` を追加する（pack-rules のテーブルに1行）。
3. zip 化時に「Codex 生成のテキストファイル」＋「staging のバイナリ」を合流させる。
   Codex はモデル JSON からテクスチャパスを参照するだけで、バイナリを生成しない。

今回この布石として実装するのは次の2点だけ：
- §4.2 のとおり白リスト等をテーブル定義に分離すること
- `writeGeneratedFiles`（テキスト書き出し）と zip 化を分離した現行2段構成を崩さないこと

## 7. リスクと割り切り

- **pack_format ずれ**：Java はバージョン毎に pack_format が変わる。Codex 生成に任せるため
  古い値になる可能性がある。`JAVA_TARGET_VERSION` で明示し、ずれたらゲーム側の警告を許容
  （読み込み自体は可能）。将来更新が必要になったら §4.2 の `JAVA_PACK_FORMATS` 対応表を
  pack-rules に追加して吸収する（置き場だけ確定済み）。
- **mcfunction の安全性**：JSON と違い構文検証ができない。危険コマンド denylist と
  白リストパス制限のみで割り切る（ローカル利用・自己責任の範囲）。
- **エディション切替でのリセット**：作業中の仕様が消える。確認ダイアログで軽減。
- **ブランチ運用**：実装は `codex-migrate-web-monorepo` 上で行う（現行正本ブランチ）。
  master への統合は design-consolidation.md の一本化作業と合わせて別途判断。
- ロールバック方針：機能はテーブルとフラグで分離されており、UI の切替を隠せば
  Bedrock 専用の現行挙動に戻る。

## 8. テスト・検証

- 単体（vitest）：
  - validateSpec：java×item / java×script(itemUse) / bedrock×resourcepack がエラーになる
  - specChecks ⇔ validateSpec の同値性（java / resourcepack を含む代表ケース）
  - pack-rules：各パック種別の白リスト通過・拒否の代表ケース
- 手動確認：
  1. Bedrock で従来どおりレシピ生成 → `.mcpack` 出力（回帰）
  2. Java でレシピ生成 → `*-datapack.zip` を `javaパックファイル` に出力 →
     zip 内に `pack.mcmeta` と `data/<ns>/recipe/*.json` があること
  3. Java で表示名変更 → `*-resourcepack.zip` → zip 内に
     `assets/minecraft/lang/ja_jp.json` と `en_us.json`
  4. WebView2 デスクトップ起動でエディション切替・生成が動くこと
  5. （任意）Java 実ゲームの `datapacks` / `resourcepacks` に配置して動作確認
