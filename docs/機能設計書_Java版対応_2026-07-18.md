# 設計書：Java版対応（データパック＋リソースパックJSONのみ）

現在 Bedrock（統合版）専用の Addon Chat Builder（`apps/web`）に、Minecraft Java Edition 向けの
データパック（.zip）とリソースパック（JSONのみの .zip）の生成機能を追加する。
Java 生成物は Codex に任せず**アプリが決定論的に生成**する。テクスチャ（PNG）同梱は今回実装しないが、
将来追加できる構造にしておく。

- **作成日**: 2026-07-18
- **対象アプリ**: addon-chat-builder-desktop（`apps/web`。WPF/WebView2 ラッパーは変更しない）
- **種別**: 機能設計書
- **対象バージョン**: apps/web v0.2.0
- **前提コミット**: 8617e49（2026-07-18 に master へマージ済み）
- **ステータス**: 実装済み・受入待ち（実装コミット `786e280`、`feature/java-edition-support`。自動検証・ZIP生成・Chrome UI確認完了、§8のJava 1.21.5実ゲーム3種確認待ち）
- **関連**: [レビュー_Java版対応設計_2026-07-18.md](レビュー_Java版対応設計_2026-07-18.md)（P1×3/P2×4/P3×2 を本版へ反映）/
  [design-consolidation.md](design-consolidation.md) / [design-ui-optimization.md](design-ui-optimization.md)

  （`mcpackPath`）に関する記述を本設計が supersede する（§4.3 の `packPath` が正本。JAVA-REV-P3-01）。

## 1. 目的・背景

- 現状は Bedrock 用アドオン（.mcpack）のみ生成できる。Java 版でも使いたい。
- 事前判断（2026-07-18）：新規アプリは作らず**既存アプリへの機能追加**とする。
  理由：チャットUI・OpenAI連携・zip化パイプラインの大半がエディション非依存で再利用でき、
  新規アプリだと .env / ランチャー / ポートの二重管理になる（YAGNI違反）。
- **実装先の確定（2026-07-18）**：ユーザーが日常利用している画面（草原テーマ＋設計図UI）は
  本リポジトリ `apps/web` であることをスクリーンショットで確認した。実装は master 上の
  `apps/web` を起点とする（旧単体版 `addon-chat-builder` には手を入れない）。
- **設計レビュー反映（2026-07-18）**：実装前レビューで「Codex 生成＋緩い検証では、ゲームが
  読み込めない zip を正常終了で出力し得る」（JAVA-REV-P1-01/02）等の指摘を受け、
  **Java 生成を決定論的テンプレート方式へ変更**した。Codex によるファイル生成は Bedrock 専用に残す。
- Java 対応範囲（ユーザー決定済み）：
  - **データパック**（レシピ・一定間隔のチャット通知）
  - **リソースパック（JSONのみ）**。テクスチャ（PNG）は将来対応。
  - Mod（Forge/Fabric）は対象外。
  - Java 版の出力先は **Java 専用フォルダ**を新設する。

## 2. スコープ

### 2.1 IN

- `AddonSpec` へのエディション概念（`edition: "bedrock" | "java"`）の追加と**サーバー側 edition ロック**
- Java 用の生成種類（すべてアプリによる決定論的生成。Codex 不使用）：
  - `recipe`（データパック：作業台の shaped レシピ。バニラ `minecraft:` アイテム限定）
  - `script`（データパック：`schedule` による**秒指定の定期チャット通知**。`intervalSeconds` を新設）
  - `resourcepack`（リソースパック：`item/block.minecraft.*` の lang 上書き。JSONのみ）
- Java 対象バージョンの**厳密指定**（サポート表による解決。未知バージョンは生成前拒否）
- 出力形式：Java は `.zip`（データパック / リソースパック）、専用出力先フォルダ
- UI：エディション切替（進行中リクエスト時は無効化・遅延応答破棄）、エディション別スターター例・
  設計図行・生成ボタン表示、対象 Java バージョン表示
- `/api/build` 応答の `packPath` への統一（供用箇所の同時変更。§4.3）
- 将来のテクスチャ同梱に備えた構造化（§4.8。実装はしない）

### 2.2 OUT（今回触らない・既存挙動を保つ）

- Bedrock 側の生成仕様（Codex 生成・白リスト検証・.mcpack 出力・既定出力先）は現行維持
- .NET WPF/WebView2 ラッパー（リポジトリ直下 `src/`）と desktop 認証
  （`verifyDesktopRequest` / `DESKTOP_API_TOKEN`）は変更しない
  （WPF は .env の任意キーを Web プロセスへ渡すため、環境変数追加のみで WPF 変更不要
  ＝レビューで裏付け済み）
- `/api/chat` の `suggestedReplies` / `recommendedReply` 応答仕様（現行維持）
- 旧単体版 `Minecraft_Addon\addon-chat-builder` は変更しない（撤去は
  [design-consolidation.md](design-consolidation.md) の別判断）
- テクスチャ（PNG）の生成・アップロード・同梱（将来対応）
- Java の Mod（Forge/Fabric）対応
- Java `script` の `itemUse` / `blockBreak` 相当イベント（将来検討）
- Java でのカスタムアイテム追加（バニラのデータパックでは不可能。§4.1 でエラーにする）
- Java 1.21.9 以降（`pack.mcmeta` が `min_format`/`max_format` 必須の新形式）への対応
  （サポート表の拡張ポイントのみ確保。§4.2）
- Bedrock のリソースパック対応
- DB導入・ポート変更

## 3. 全体方針

- **エディションは仕様（AddonSpec）の属性**とし、チャット整理 → 仕様確定 → 生成 → zip化 の
  フローはエディション共通のまま維持する。
- **生成方式はエディションで分ける**（JAVA-REV-P1-02 反映）：
  - Bedrock：現行どおり Codex 生成＋白リスト・JSON・スクリプト検証（変更なし）
  - Java：今回スコープは全て構造化済みのため、**`AddonSpec` からアプリが決定論的に全ファイルを
    生成**する（新規 `java-generator.ts`）。LLM の揺らぎ・形式ずれが原理的に発生せず、
    白リスト検証や mcfunction denylist も Java には不要になる。
- エディション・パック種別ごとの規則（出力サフィックス・Java バージョン表・Bedrock 白リスト）は
  テーブル定義（新規 `pack-rules.ts`）に分離し、分岐を散らさない。
- Java のパック種別は kind から自動決定する：`recipe` / `script` → データパック、
  `resourcepack` → リソースパック。ユーザーにパック種別を直接選ばせない（1ビルド＝1パック）。
- **UI が唯一のエディション決定元**（JAVA-REV-P2-01 反映）：サーバーは UI から受けた edition を
  ロック値として扱い、モデル出力の edition を常に上書きする。
- 現行UIの不変条件 **`specChecks(spec).every(ok) === (validateSpec(spec).length === 0)`**
  （`tests/unit/addon-view.test.ts` で担保）は拡張後も維持する。

## 4. 詳細仕様

### 4.1 スペックモデル（apps/web/src/lib/spec.ts）

```ts
export type Edition = "bedrock" | "java";
export type AddonKind = "recipe" | "item" | "script" | "resourcepack";

export type AddonSpec = {
  edition: Edition;          // 追加。既定 "bedrock"
  // ...既存フィールドは維持
  script?: {
    event: "itemUse" | "blockBreak" | "interval";
    summary: string;
    message: string;
    intervalSeconds?: number; // 追加。java × interval で必須（5〜3600）。bedrock では未使用
  };
  resourcepack?: {           // 追加（Java専用）。pack説明はトップレベル description を再利用（P3-02）
    langEntries: { key: string; value: string }[];
  };
};
```

- エディション×kind の対応表：

| kind | bedrock | java |
|---|---|---|
| recipe | ○（現行） | ○ データパック（バニラ `minecraft:` アイテム限定） |
| item | ○（現行） | ×（バニラデータパックでは不可。validateSpec でエラー文言を返す） |
| script | ○（現行 itemUse/blockBreak/interval） | △ データパック。`interval` のみ・`intervalSeconds` 必須 |
| resourcepack | ×（今回対象外） | ○ JSONのみ（lang 上書き） |

- `validateSpec` に追加する検証：
  - `edition` が不正 → エラー
  - `java` × `item` → 「Java版ではアイテム追加はデータパックで実現できません（Mod が必要）」
  - `java` × `script` で `interval` 以外 → 「Java版の script は interval のみ対応です」
  - `java` × `script` で `intervalSeconds` が未設定または 5〜3600 の範囲外 → エラー
    （既定値は 60。チャット整理時にモデルが未指定なら 60 を提案する）
  - `bedrock` × `resourcepack` → 「リソースパックは Java版のみ対応です」
  - `java` × `recipe`：`resultItem` と `key` の全アイテムが `minecraft:` 接頭辞であること
    （バニラ限定。JAVA-REV 確定事項3）
  - `resourcepack`：`langEntries` 1件以上。key は `^(item|block)\.minecraft\.[a-z0-9_]+$`、
    value は trim 後非空、key の重複は拒否（JAVA-REV-P3-02）
- `specChecks` も同じ内容で拡張する（§3 の不変条件を維持）：
  - `java` × `script`：「間隔」行（`intervalSeconds` 秒）を追加
  - `resourcepack`：「表示名の変更」行（n件）を追加
  - エディションは UI 切替で常に確定しているため、ブロッキング行にはしない
    （設計図には §4.7 の情報行として表示）

### 4.2 パックルール（新規 apps/web/src/lib/pack-rules.ts）

```ts
export type PackType = "mcpack" | "datapack" | "resourcepack";

export function resolvePackType(spec: AddonSpec): PackType;
// bedrock → "mcpack" / java × recipe・script → "datapack" / java × resourcepack → "resourcepack"

export const OUTPUT_SUFFIX: Record<PackType, string>;
// ".mcpack" / "-datapack.zip" / "-resourcepack.zip"

export const BEDROCK_ALLOWED_PATHS: RegExp[];   // 現行白リストを移設（bedrockのみで使用）

// Java 対象バージョンのサポート表（JAVA-REV-P1-01 反映）
export type JavaVersionRule = {
  datapackFormat: number;      // データパックの pack_format
  resourcepackFormat: number;  // リソースパックの pack_format
  plainIngredients: boolean;   // レシピ材料の表記（24w33a 以降は文字列形式）
};
export const JAVA_VERSIONS: Record<string, JavaVersionRule> = {
  "1.21":   { datapackFormat: 48, resourcepackFormat: 34, plainIngredients: false },
  "1.21.4": { datapackFormat: 61, resourcepackFormat: 46, plainIngredients: true },
  "1.21.5": { datapackFormat: 71, resourcepackFormat: 55, plainIngredients: true },
  "1.21.7": { datapackFormat: 81, resourcepackFormat: 64, plainIngredients: true }
};
export function resolveJavaVersion(): { version: string; rule: JavaVersionRule };
// env JAVA_TARGET_VERSION（既定 "1.21.7"）→ JAVA_VERSIONS を引く。
// 表に無いバージョンは【生成前にエラーで拒否】（Codexに選ばせない・黙って古い値を使わない）
```

- `pack_format` はパック種別で値が異なるため**必ず種別別に持つ**。「1.21系」のような曖昧指定は
  受け付けない（env には表のキーを厳密指定。既定 `1.21.7`）。
- Java 1.21.9 以降で `pack.mcmeta` が `min_format`/`max_format` 必須の新形式になるため、
  対応時は `JavaVersionRule` にメタ形式世代を追加し `java-generator.ts` の pack.mcmeta 生成を
  拡張する（今回は対象外。§2.2）。
- 将来の Java 更新はこの表への 1 行追加で吸収する（ユーザー確定事項2「更新可能性の考慮」）。

### 4.3 生成パイプライン（apps/web/src/lib/addon-generator.ts）

- `buildMcpack` を `buildPack` に一般化する。戻り値は `{ packPath, files }`。
  - **`mcpackPath` → `packPath` 改名の根拠（JAVA-REV-P3-01）**：リポジトリ内の利用箇所は
    `apps/web/src/app/page.tsx`（BuildResult 型・結果表示）のみで、WPF は応答を経由しない
    （grep で確認済み）。外部利用者がいないため alias は置かず、page.tsx を同時変更する。
    `design-ui-optimization.md` の該当記述は本設計が supersede する（冒頭に明記済み）。
- エディションで生成器を分岐：
  - bedrock：`generateAddonFilesWithCodex(spec)` → 現行の白リスト・JSON・スクリプト検証
    （`BEDROCK_ALLOWED_PATHS` 参照に変えるのみで挙動不変）
  - java：`generateJavaFiles(spec)`（§4.4）→ 検証は生成器のユニットテストで担保
    （生成器がファイルの唯一の出所のため、実行時白リストは適用しない。
    mcfunction denylist は不要＝生成コマンドが `tellraw`/`schedule` の固定形のみ。JAVA-REV-P2-03）
- zip 化は現行どおり AdmZip。出力ファイル名は `outputName + OUTPUT_SUFFIX[packType]`
  （例：`my-addon-datapack.zip`）。

### 4.4 Java 決定論的生成（新規 apps/web/src/lib/java-generator.ts）

`AddonSpec` と `resolveJavaVersion()` から、以下を**固定テンプレートで**生成する
（JAVA-REV-P1-01/02/03・P2-03 反映）。JSON はすべて `JSON.stringify` で構築する
（message 等のエスケープ漏れを排除）。

**共通**
- `pack.mcmeta`：アプリが生成（Codex に選ばせない）。
  `{"pack":{"pack_format":<種別別の値>,"description":<spec.description>}}`
- `README.txt`：導入手順（datapacks / resourcepacks フォルダへの配置方法）と対象バージョンを記載。

**datapack × recipe**
- `data/<namespace>/recipe/<outputName>.json`：`minecraft:crafting_shaped`。
  - `plainIngredients: true` のバージョン：key の値は `"minecraft:diamond"` の文字列形式
  - `false` のバージョン：`{"item":"minecraft:diamond"}` 形式
  - `result` は `{"id":<resultItem>,"count":<resultCount>}`
- 材料・完成品はバニラIDのみ（§4.1 で検証済み）。**ID の実在確認は行わない**割り切り：
  ID一覧の同梱は保守コストに見合わない（YAGNI）。存在しないIDのレシピはゲーム側で無効になる旨を
  README.txt に記載し、チャット整理プロンプトで正しいバニラIDを使うよう指示する。

**datapack × script（interval）**
- `data/<namespace>/function/load.mcfunction`：
  `schedule function <namespace>:main <intervalSeconds>s replace`
- `data/<namespace>/function/main.mcfunction`：
  `tellraw @a <JSON.stringify({text: message})>` ＋ 末尾で
  `schedule function <namespace>:main <intervalSeconds>s replace`（自己再予約）
- `data/minecraft/tags/function/load.json`：`{"values":["<namespace>:load"]}`
- **`tick.json` は使わない**（毎 tick＝毎秒20回の実行を構造的に排除。JAVA-REV-P1-03）。

**resourcepack**
- `assets/minecraft/lang/ja_jp.json` と `assets/minecraft/lang/en_us.json`：
  `langEntries` から**同一内容**で生成（両ファイルの entry 集合一致をテストで担保）。

### 4.5 出力先（apps/web/src/lib/paths.ts / .env）

- エディション別の既定出力先：
  - bedrock：現行維持（`DEFAULT_OUTPUT_DIR` → 無指定時 `<cwd>\mcpack`）
  - java：`DEFAULT_OUTPUT_DIR_JAVA` → 無指定時 `<cwd>\javapack`
- 運用上の配布置き場（.env で指定・確定済み）：
  - bedrock：`D:\my-app2\Minecraft_Addon\mcpackファイル`（現行）
  - java：`D:\my-app2\Minecraft_Addon\javaパックファイル`（新設）
- **zip の Git 管理方針（JAVA-REV 確定事項4）**：`javaパックファイル\*.zip` は
  `mcpackファイル\*.mcpack` と同一運用とし、**親リポジトリ D:\my-app2 で Git 管理（コミット）する**
  （mcpack の既存実績に合わせる。除外設定は追加しない）。
- UI の「既定に戻す」はエディションに応じた既定へ戻す。

### 4.6 OpenAI 連携（apps/web/src/lib/openai.ts）

- チャット整理（refineAddonSpec）は**両エディションで使用**（仕様の対話整理のみ。ファイル生成はしない）：
  - リクエストに UI 選択の `edition` を**ロック値**として追加。`normalizeRawSpec` で
    モデル出力の edition を**常にロック値で上書き**し、edition・kind の非対象セクション
    （java なら item、bedrock なら resourcepack 等）を消去する（JAVA-REV-P2-01）。
  - JSON スキーマに `edition`・`resourcepack`・`script.intervalSeconds` を追加
    （strict スキーマのため全項目必須・未使用時は安全な既定値。現行方式踏襲）。
  - `suggestedReplies` / `recommendedReply` の応答は現行のまま維持する。
  - Java 用 system プロンプト：「対応範囲は recipe / script(interval) / resourcepack(langEntries)」
    「item は不可。代わりに resourcepack の表示名変更か統合版を提案」「レシピのIDは
    `minecraft:` のバニラIDのみ」「interval は秒指定（5〜3600、既定60）で聞き取る」
    「lang の key は item/block.minecraft.* のみ」を明記。
- Codex 生成（generateAddonFilesWithCodex）は **Bedrock 専用**とし、プロンプト・スキーマは
  現行から変更しない（java パスからは呼ばれない）。
- 環境変数 `JAVA_TARGET_VERSION`（既定 `1.21.7`）。値の解決と検証は `pack-rules.ts`（§4.2）。

### 4.7 UI（apps/web/src/app/page.tsx・styles.css・src/utils/addon-view.ts・新規 /api/config）

現行UI（草原テーマ＋「チャット＋ライブ設計図」）の構造は維持し、以下を追加・切替する。

- **トップバー**：`topbar-right` に**エディション切替**（ブロック調セグメント：統合版 / Java版）。
  eyebrow は `MINECRAFT BEDROCK` / `MINECRAFT JAVA EDITION` に連動。
- **切替時の挙動（JAVA-REV-P2-02 反映）**：
  - `isChatting || isBuilding || isSelectingOutput` の間は**切替ボタンを無効化**する
  - 加えて世代カウンタ（generation id）を持ち、chat / build / select-folder の応答受信時に
    発行時の世代と現在世代が一致しない結果は**破棄**する（遅延応答による新エディション状態の
    上書きを防ぐ）
  - 切替実行時は会話・仕様・生成結果をリセット。作成開始後（hasStarted）は確認ダイアログを出す
- **対象バージョンの表示**：新規 `GET /api/config` が `{ javaTargetVersion }` を返し
  （`resolveJavaVersion()` と同一の解決値。未サポート値なら起動直後にエラー表示）、
  設計図のエディション行に「Java版（1.21.7）」のように表示する。
- **スターター例**をエディション別に切替：
  - Java 例：「新しい武器レシピを作りたい」「アイテムの表示名を変えたい（リソースパック）」
    「1分ごとにチャットへメッセージを流したい」
  - 「新しいアイテムを追加したい」は Java では出さない
- **設計図（BLUEPRINT）**：
  - 先頭に**エディション行**（情報行、「統合版」/「Java版（<対象バージョン>）」）
  - Java 時は**パック種別行**（情報行、「データパック」/「リソースパック」）
  - `kindLabel` に `resourcepack` → 「リソースパック」を追加
  - `java` × `script` は「間隔」行、`resourcepack` は「表示名の変更 n件」行（§4.1 の specChecks に対応）
- **生成ボタン（ready-cta と設計図下部の両方）**の文言をパック種別で切替：
  `.mcpack を生成` / `データパック(.zip) を生成` / `リソースパック(.zip) を生成`
- **出力先ボックス**：ラベル・既定値をエディション連動（§4.5）。
- 生成完了メッセージ・result-box の `BuildResult.packPath` 追従（§4.3）。

### 4.8 将来のテクスチャ対応への布石（今回は実装しない）

1. PNG アップロード API（例 `/api/assets`）＋セッション毎 staging フォルダ

## 7. リスクと割り切り

- **バージョン表の保守**：Java 本体更新時は `JAVA_VERSIONS` への行追加が必要。未知バージョンは
  生成前に明示的エラーで止まる（黙って古い形式で出力しない）ため、壊れた zip は出ない。
  1.21.9 以降の pack.mcmeta 新形式は対象外と明記（§2.2）。
- **レシピIDの実在確認はしない**：バニラID一覧は同梱しない（YAGNI）。`minecraft:` 接頭辞の
  形式検証＋プロンプト誘導＋README.txt の注意書きで割り切る。存在しないIDでもゲームは
  クラッシュせず該当レシピが無効になるだけであることを確認済みの前提とする（§8 実機確認で検証）。
- **エディション切替でのリセット**：作業中の仕様が消える。進行中無効化＋確認ダイアログで軽減。
- **`packPath` 改名**：外部利用者なし（§4.3 で根拠記録）。page.tsx と同一コミットで変更するため
  中間状態は存在しない。
- ロールバック方針：Java 生成は独立モジュール（`java-generator.ts`・`pack-rules.ts`）で、
  UI の切替を隠せば Bedrock 専用の現行挙動に戻る。

## 8. テスト・検証

**単体（vitest）**

- `java-generator`：3種それぞれについて、代表 spec からの**生成ファイル全文の完全一致**テスト
  （pack.mcmeta の pack_format がパック種別・バージョン別に正しいこと、
  1.21 と 1.21.7 の材料表記差、tellraw の JSON エスケープ、load.json の参照先一致、
  ja_jp / en_us の entry 集合一致を含む）
- `pack-rules`：`resolveJavaVersion` の既定値・env 指定・**未知バージョン拒否**。
  `resolvePackType`・OUTPUT_SUFFIX の対応
- `spec`：validateSpec（java×item / java×script(itemUse) / intervalSeconds 境界 4・5・3600・3601 /
  bedrock×resourcepack / minecraft: 接頭辞違反 / langEntries の重複key・空value・不正key）
- `spec × addon-view`：specChecks ⇔ validateSpec の同値性（java / resourcepack を含む代表ケース）
- `openai`（normalizeRawSpec）：モデル出力の edition がロック値と不一致でも**ロック値で上書き**され、
  非対象セクションが消去されること

**手動確認（受け入れゲート）**

1. Bedrock で従来どおりレシピ生成 → `.mcpack` 出力（回帰）
2. Java でレシピ生成 → `*-datapack.zip` の中身が §4.4 のレイアウトと一致
3. Java で定期メッセージ（例 60秒）→ zip 内の mcfunction / load.json が §4.4 と一致
4. Java で表示名変更 → `*-resourcepack.zip` に ja_jp / en_us 両方
5. WebView2 デスクトップ起動でエディション切替・生成が動く。chat 応答待ち中に切替ボタンが
   無効化されている
6. **実ゲーム読込確認（必須。JAVA-REV-P1-02）**：対象バージョンの Java Edition 実機で
   3種すべてを確認する——(a) レシピが作業台でクラフト可能、(b) 定期メッセージが指定間隔で
   チャットに出る（毎tick連打しない）、(c) 表示名が変わる。データパックは `/datapack list` で
