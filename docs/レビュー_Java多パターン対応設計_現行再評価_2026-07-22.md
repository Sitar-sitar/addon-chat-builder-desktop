## 1. 判定

**No-Go（現行のまま正式受入・設計確定不可。P1 2件、P2 5件、P3 1件）**。

26.2 の Pack Format、1.21.9+ の `min_format` / `max_format`、26.1+ の `minecraft:time_check.clock`、cooking result の count 対応という版差の選定自体は、2026-07-22 時点の公式情報と整合している。capability カタログ、edition 分離、client/server build gate、決定論的テンプレートという大枠も妥当である。

しかし、本設計の出発点である「未対応の意図を別パターンへ黙って劣化させない」が `/api/build` の実行時境界でまだ成立していない。また、設計は各 Phase の実ゲーム受入後に公開すると定める一方、Phase 1〜5 の新代表ケースが未実施のまま `RELEASED_JAVA_PHASE = 5` で全機能が公開されている。したがって、自動テスト 38 件の合格だけでは本設計の受入条件を満たしたとは判定できない。

- [Minecraft Snapshot 17w43b 公式記事（Data Pack Usage & Installation）](https://www.minecraft.net/nl-nl/article/minecraft-snapshot-17w43a): data pack は ZIP またはディレクトリを `<world>/datapacks/` に置く。P2-03 の根拠。

## 4. 確認できた妥当点

- Bedrock / Java の discriminated union 分離、Bedrock Codex 入力の明示射影は、既存挙動の保護方針として妥当。
- capability の phase / 版要件をカタログへ集約し、client と server が独立に解決する方針は妥当。
- 26.2 Pack Format、1.21.9+ mcmeta、26.1+ predicate clock、cooking count の表駆動化は公式仕様と整合。
- advancement revoke / scoreboard reset を condition 不成立時も無条件実行する設計は、遅延発火を残さない。
- 現行 `npm run lint` は合格。Vitest は sandbox 内で `spawn EPERM` になったが、同一作業ツリーを制限外で再実行し 8 files / 38 tests pass を確認した。
