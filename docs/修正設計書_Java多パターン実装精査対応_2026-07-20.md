## 1. 目的・背景

2026-07-20 の実装コード精査（実装ログ Phase 2 の成果物レビュー）で以下を検出した。

- **M1（必須・不具合）**: `validateJavaRecipe`（`apps/web/src/lib/spec.ts`）の shaped レシピ検証は「pattern の記号 ⊆ key」しか確認していない。逆方向の「key の記号 ⊆ pattern の記号」が未検証のため、LLM が pattern で使わない keyEntries を返すと検証を素通りし、Minecraft Java 実機が「Key defines symbols that aren't used in pattern」でレシピ読み込みを拒否する**無効な ZIP が生成**される。v0.2.0 実装レビュー P2-01（`44bc3e3`）で行った pattern/key 整合検証の残り漏れ。
  - 再現条件: shaped レシピ spec の `key` に pattern 未使用の記号（例: pattern `["AA"]`, key `{A: minecraft:stick, B: minecraft:iron_ingot}`）を与えて build する。現状は ZIP 生成に成功するが、ゲーム内でレシピが無効になる。
- **M2（保守）**: capability 判定ロジックが `capabilitiesForSpec`（`pattern-catalog.ts`）と `javaCapabilitiesForValidation`（`spec.ts`）に二重実装されており、既に挙動が食い違っている（script で `javaScript` 未設定時、前者は `[]`、後者は `resourcepack.lang` へフォールスルー）。capability 追加時に片方だけ直す事故の温床。
- **M3（保守）**: `fetchOpenAiResponse`（`openai.ts`）の `fetch` にタイムアウトがなく、OpenAI API がハングするとサーバ側リクエストが無期限に待つ。
- **M4（保守）**: `java-generator.ts` の `JAVA_GENERATOR_HANDLERS[capability];` は実行時 no-op の式文（型網羅は `Record<JavaCapabilityId, true>` の型定義だけで担保済み）。
- **M5（保守）**: 本ツールで既定 namespace（`my_addon` 等）のまま生成した複数のデータパックを同一ワールドへ導入すると function パスが衝突し、後勝ちで沈黙の上書きになる。README.txt に注意書きがない。

## 3. 全体方針

- いずれも小さな局所修正で、機能追加はしない。M1 は既存の shaped 検証ブロックへの 1 条件追加、M2〜M4 は削除・置換中心で差分を最小にする。
- M2 は `spec.ts` 側の `javaCapabilitiesForValidation` を削除し、`pattern-catalog.ts` の `capabilitiesForSpec` を import して使う。`pattern-catalog.ts` → `spec.ts` の依存は type-only（実行時に消える）のため、逆方向の実行時 import を足しても循環しない。逆向きの集約（spec 側へ寄せる）は、生成側 `java-generator.ts` が既に `capabilitiesForSpec` を使っているため採らない。
- M3 は `AbortSignal.timeout()` を使う（Node 18+ / Next.js 環境で利用可）。リトライ機構は足さない（既存のフォールバックモデル逐次試行がそのまま次候補へ進む）。タイムアウトは**モデル候補1件ごと**にリセットする（後述 M3 の確定事項）。
- M4 は no-op 削除に限定し、網羅性は「ID 登録漏れの検知（型）」と「版×capability の期待ファイル／内容テスト（実装網羅）」の二段構えとする。真偽値表を実 handler map へ格上げする案は、生成が既に kind 別手書き分岐で成立しており過剰実装のため採らない（KISS）。

## 4. 詳細仕様

### M1: 未使用 key 記号の検証（spec.ts）

`validateJavaRecipe` の shaped 分岐で、pattern から抽出済みの `symbols` 集合を使い、逆方向チェックを追加する。

```ts
if (Object.keys(r.key).some((s) => !symbols.has(s)))
  errors.push(
    "Java版のレシピ素材に、パターンで使われていない記号があります。",
  );
```

- `symbols` は既存実装どおり空白を除外して抽出する（空白は Minecraft の予約記号。`normalizeJavaRawSpec` が空白記号の keyEntries を除去するため key 側に空白は来ない）。
- pattern が空白のみで key が 1 件以上ある場合も、このチェックで拒否される（実機でも無効なため正しい挙動）。
- エラー文言は既存の「Java版のレシピ〜」系に合わせる。

### M2: capability 判定の一本化（spec.ts / pattern-catalog.ts）

- `spec.ts` の `javaCapabilitiesForValidation` を削除し、`validateJava` は `pattern-catalog.ts` の `capabilitiesForSpec` を呼ぶ。
- 挙動差分は「script かつ `javaScript` 未設定」のときのみ: 旧実装は `resourcepack.lang` を要求していたが、新実装は `[]`（要求なし）になる。この場合も `validateJavaScript` が「Javaスクリプト情報が未設定です。」を必ず返すため、**build 可否は変わらない**。
- `pattern-catalog.ts` 側は変更しない（`capabilitiesForSpec` が正本）。

### M3: OpenAI fetch タイムアウト（openai.ts）

- `fetchOpenAiResponse` の `fetch` に `signal: AbortSignal.timeout(120_000)` を付与する（chat / Codex 共通、固定 120 秒）。
- **確定事項（REV-P2-03）**: タイムアウトの粒度は**モデル候補1件ごと**。`AbortSignal.timeout()` は `for (const model of models)` ループ内の各 `fetch` 呼び出し時点で生成し、候補ごとに新しい signal を割り当てる。これにより「primary が 120 秒でタイムアウト → 次の fallback を新しい signal で再試行」が成立し、最初の 120 秒経過で後続候補まで即 abort されることはない。「要求全体で 120 秒」ではない。
- タイムアウト時は既存の catch に落ち、フォールバックモデルがあれば次候補を試行、なければ「OpenAI API エラー / Codex API エラー: <理由>」で失敗する（既存のエラー整形を踏襲）。
- **自動テスト（REV-P2-03）**: Vitest の fake timer と `fetch` mock で、待ち時間ゼロで以下を検証する。(a) primary の timeout 後、fallback が新しい signal で試行される、(b) 全候補失敗時に API 種別付き（`OpenAI API エラー` / `Codex API エラー`）のエラーになる、(c) chat と Codex がともに共通 helper を通ることを 1 件ずつ確認する。

### M4: no-op 式文の削除・網羅定数の役割訂正（java-generator.ts）

- `generateJavaFiles` 内の `JAVA_GENERATOR_HANDLERS[capability];` の行を削除する。
- **役割訂正（REV-P2-01）**: `JAVA_GENERATOR_HANDLERS: Record<JavaCapabilityId, true>` は真偽値表であり、handler 実装の網羅（対応する生成物が実装済みか）は保証しない。実生成は `generateJavaFiles` の kind 別手書き分岐で行うため、ID と `true` だけを追加しても生成が未実装/別種のまま型は通り得る。したがってこの定数は「`JavaCapabilityId` の**ID 登録漏れ**を追加時にコンパイルエラーで検知する」役割に限定し、名称を `JAVA_GENERATOR_CAPABILITY_COVERAGE` へ改名して意図を明示する。**実装網羅は §8 の版×capability の期待ファイル／内容テストで担保する**（真偽値表では担保しない）。実 handler map（値を生成関数にして dispatch もそこを通す）へ格上げする案は採らない（KISS、§3 全体方針参照）。
- 併存する `spec.ts` の `JAVA_CAPABILITY_VALIDATORS`（同型の真偽値表）は本レビューのスコープ外につき本修正では触れない。

### M5: README.txt への namespace 衝突注意（java-generator.ts）

- **警告対象の限定（REV-P2-04）**: 生成パスの実衝突条件は kind ごとに異なる。script は `data/<namespace>/function/load.mcfunction` 等の**固定名**のため namespace 一致で必ず衝突する。recipe は `data/<namespace>/recipe/<outputName>.json` で namespace＋outputName、loot は namespace を使わず `data/minecraft/loot_table/blocks/<targetBlockId>` で衝突する。全データパックへ「同じ namespace で上書き」と出すと、recipe へ過剰警告し loot では namespace を変えれば安全だと誤読させる。
- よって今回の注記は **`spec.kind === "script"` のデータパックに限定**する（namespace/function 衝突のみを扱う。KISS）。recipe・loot の衝突注意は本修正では扱わない（別 kind の実パス単位注記は将来必要になったら別途）。
- `buildReadme` で、`spec.kind === "script"` の場合のみ notes に追記する:

```
注意: 同じ namespace（<spec.namespace>）を使う別のスクリプト系データパックと同時に導入すると、機能が上書きされて正しく動かないことがあります。
```

- recipe・loot・リソースパックには追記しない。

## 7. リスクと割り切り

- M1 で、これまで「生成できてしまっていた」不正 spec が build 拒否に変わる（意図した仕様厳格化。ゲーム内で無効になる ZIP を出すより良い）。境界は `validateSpec` 単体ではなく `buildPack` の実ゲート（§8 の build 境界テスト）で固定する。
- M3 の 120 秒は実測に基づかない安全側の固定値・**候補1件ごと**。長考モデルで不足が判明したら環境変数化を後続で検討（今回は YAGNI で固定）。異常系は fake timer ＋ fetch mock の自動テストで検証する（目視のみに依存しない）。
- M4 は網羅定数を改名し役割を「ID 登録漏れ検知」に限定する。実装網羅は版×capability テストが正本であり、真偽値表には期待しない。`JAVA_GENERATOR_HANDLERS` を参照する箇所（テスト含む）は改名に追随させる。
- M5 は script 系データパックの README 文言追加のみ。recipe（namespace＋outputName）・loot（targetBlockId）の衝突は今回扱わず、衝突の検出・防止も行わない（正本 namespace 管理は本ツールの責務外と割り切る）。
- ロールバックは revert 1 コミットで可能（生成物への影響は script の README 1 行のみ）。
